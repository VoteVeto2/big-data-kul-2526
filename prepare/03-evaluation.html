The central thesis: it's all about generalization

Restate the problem: why can't we just measure how well a model fits the data it was trained on?
Because fitting the training data is trivially easy and tells you nothing about the future. A fully-grown decision tree, a 1-nearest-neighbour model, or literally a database lookup all achieve 100% training accuracy by memorizing. The real question is whether the model works on unseen data it has never touched — that is generalization, and the only way to measure it is a held-out test set.

The problem, and the competing branch

Generalization is how well a model performs on unseen data — future, noisy, robust conditions. The training set is also called the development set: the data you're allowed to do anything with. Measuring performance there gives only a very initial sanity check, because perfect fit is free.

Drill the why — why is 100% train accuracy worthless?

Three ways to "win" the training set with zero learning: a tree grown until every leaf is pure, a \(k\)-NN with \(k=1\) (each point is its own neighbour), or a database that just stores and looks up. None of these learned a pattern; they memorized. So a high training score is consistent with both a great model and a useless one — it cannot distinguish them. Only out-of-sample evaluation can.

There is a competing branch — the statisticians' alternative: restrict yourself to simple, linear, parametric models and report goodness of fit. This sidesteps the generalization problem by capping complexity, but it is not a real predictive test — you've assumed away the hard part rather than measured it.

ELI-intern · the "single-parameter elephant"

Boué (2019) showed that any dataset — an elephant, a chicken, a turtle drawing — can be approximated by a continuous differentiable function with a single real parameter \(\alpha\):

\[ f_\alpha(x) = \sin^{2}\!\big( 2^{x\tau}\cdot \arcsin\sqrt{\alpha}\,\big) \]

The moral: fitting is easy and meaningless. If one parameter can trace an elephant, your model fitting the training data perfectly proves nothing. Only a strict, challenging held-out test does.

The solution and its design decisions

  • Split first. Separate train and test before any preprocessing/featurization — otherwise statistics leak from test into train.
  • Stratify. Keep the same class distribution in train and test, so the test set is representative.
  • Prefer out-of-time when possible (more on this next).
  • The goal is to catch failure cases. Make the test set as challenging as you can — the aim is not a flattering number, it's to find where the model breaks before reality does.

Out-of-sample vs out-of-time validation

Out-of-sample

Train and test from the same time period, different instances. Test points are scattered among the train points.

Out-of-time

Test comes from a later period than train (train on the left/earlier, test on the right/later on the time axis).

Why it matters

Out-of-time is appealing because "time will always continue to progress as your model is being used" — production data is future data. The honest trade-off: you can never foresee everything, you're limited to the data you have now. The duty is to be as thorough as possible, not to be omniscient.

The confusion matrix & threshold-dependent metrics

The 2×2 table and the threshold

Models output probabilities/scores; a threshold (e.g. 0.50) turns them into labels: score \(\ge t \Rightarrow\) "yes". The confusion matrix cross-tabulates true vs predicted. Every metric below depends on the threshold, because the matrix itself does.

Memorize each metric by its denominator — that is the whole game:

MetricFormulaPlain language
Accuracy\(\dfrac{TP+TN}{\text{total}}\)fraction correct overall
Recall = Sensitivity = TPR\(\dfrac{TP}{TP+FN}\)of the actual positives, how many did we catch?
Precision = PPV\(\dfrac{TP}{TP+FP}\)of our positive predictions, how many were right?
Specificity = TNR\(\dfrac{TN}{TN+FP}\)of the actual negatives, how many did we flag correctly?
FPR = Fall-out\(\dfrac{FP}{TN+FP} = 1-\text{Spec}\)negatives wrongly called positive
Balanced accuracy\(\dfrac{\text{recall}+\text{specificity}}{2}\)average per-class recall
F1\(\dfrac{2\,PR}{P+R}\)harmonic mean of precision & recall
Worked numbers on the 3 / 2 / 2 / 7 matrix (click)
  • Accuracy \(= (3+7)/14 = \mathbf{0.71}\)
  • Recall \(= 7/(7+2) = 7/9 = \mathbf{0.78}\)
  • Precision \(= 7/(7+2) = 7/9 = \mathbf{0.78}\) (here \(FP=FN=2\), so both equal \(7/9\))
  • Specificity \(= 3/(3+2) = \mathbf{0.60}\)
  • Balanced accuracy \(= (0.78+0.60)/2 = \mathbf{0.69}\)
  • F1 \(= 2(0.78\cdot0.78)/(0.78+0.78) = \mathbf{0.78}\)
Drill the why — why harmonic mean for F1?

The harmonic mean is dominated by the smaller of the two numbers. If precision is 0.95 but recall is 0.05, the arithmetic mean is a flattering 0.50, but the harmonic mean (F1 \(\approx 0.095\)) screams "this model is useless at one of the two jobs." F1 refuses to let a model hide a catastrophic weakness behind a strong other half.

  1. It misses 70% of spam emails.
  2. 30% of all emails are spam.
  3. It catches almost all spam, but 70% of what it flags is actually legitimate mail.

Tuning the threshold

For each candidate threshold \(t\) you get a different matrix and different metrics. Sweep \(t\) and plot the metric vs cutoff. Accuracy-vs-cutoff is non-monotone (peaks mid-range, ~0.3 in the deck example). The precision/recall pair moves predictably:

  • As the threshold rises, you call fewer things positive ⇒ precision generally rises, recall generally falls.
  • At a very low threshold, recall starts at 1 (you catch everything) but precision is poor; at a high threshold, precision \(\to 1\) but recall collapses.

This connects to well-calibrated classifiers — probabilities that actually mean what they say, so a threshold of 0.7 corresponds to a real 70% likelihood.

Class imbalance: never trust accuracy alone

Restate the problem: in a fraud dataset where 1% of cases are fraud, why is a 99%-accurate model possibly useless?
Because predicting "never fraud" for everyone scores 99% accuracy (it's right on all 99% of the legitimate cases) while catching zero fraud. Accuracy rewards the majority class, so under imbalance it is trivially gamed. "Never believe reports that only mention accuracy."

Out of the box, an accuracy-optimizing model on a 1% problem predicts everything negative: \(TN=99,\ FN=1,\ FP=0,\ TP=0\) → 99% accuracy, 0 positives caught. The naïve fix — "just pick a stricter threshold" — doesn't reliably help. The real question is structural: how do I tell the model I'm willing to make some negative-side mistakes in order to catch the positives?

The levers you can pull

Many traditional models inherently optimize accuracy on the train set, but you usually care about something else. Options to bend the model toward your metric:

  • Tune the threshold on your metric of interest
  • Adjust model parameters, or the target definition
  • Sample/filter the dataset
  • Apply misclassification costs or instance weighting (easy hack: duplicate instances)
  • Adjust the loss function (if supported) — and above all, evaluate on what matters to you

Misclassification costs & cost-sensitive learning

The cost matrix and expected loss

A cost matrix \(C(\text{predicted}, \text{actual})\) assigns a penalty to each error type. Example:

cost matrix\[ \begin{array}{c|cc} & \text{Actual }0 & \text{Actual }1 \\ \hline \text{Pred }0 & C(0,0)=0 & C(0,1)=5 \\ \text{Pred }1 & C(1,0)=1 & C(1,1)=0 \end{array} \]

Read it as: "missing a positive (predicting 0 when it's really 1) is 5× as bad as a false alarm." Costs come from real observed averages (hard to get), expert judgement, or the inverse class distribution.

The expected loss of classifying \(x\) as class \(j\):

expected loss\[ l(x,j) = \sum_{k} p(k\mid x)\, C(j,k) \]

For binary, with the example costs: \(l(x,0)=p(1\mid x)\,C(0,1)\) and \(l(x,1)=p(0\mid x)\,C(1,0)\). The decision rule: classify positive iff \(l(x,1) \lt l(x,0)\).

Worked example — \(p(1\mid x)=0.22,\ C(0,1)=5,\ C(1,0)=1\)
\[ l(x,0) = 0.78\cdot 0 + 0.22\cdot 5 = 1.10 \qquad l(x,1) = 0.78\cdot 1 + 0.22\cdot 0 = 0.78 \]

\(0.78 \lt 1.10\) → classify as positive, even though \(p=0.22\) is well below 0.5. The high cost of missing a positive pulls the decision over.

The cost-sensitive threshold \(T_{CS}\)

Set \(l(x,1)=l(x,0)\) and solve for the boundary probability:

cost-sensitive threshold\[ T_{CS} = \frac{C(1,0)-C(0,0)}{\big(C(1,0)-C(0,0)\big) + \big(C(0,1)-C(1,1)\big)} \]
  • Symmetric unit costs \((C(1,0)=C(0,1)=1,\ C(0,0)=C(1,1)=0)\): \(T_{CS} = \tfrac{1}{1+1} = \mathbf{0.5}\) — the default.
  • \(C(0,1)=5,\ C(1,0)=1\): \(T_{CS} = \tfrac{1}{1+5} = \mathbf{0.1667}\). Since \(0.1667 \le 0.22\), classify positive — same decision as the expected-loss calculation.
Drill the why — why does a higher cost of missing positives lower the threshold?

If false negatives are expensive, you want to be quicker to call something positive — so you lower the bar a score must clear. Cost on the positive side and the decision threshold move in opposite directions: expensive misses ⇒ low threshold ⇒ more positives flagged. \(T_{CS}\) is just the algebra of "where do the two expected losses tie?"

Resampling for imbalance: sampling ⇔ cost-sensitivity

The key equivalence (exam favourite)

"Sampling the dataset so the minority class equals the majority class boils down to biasing the classifier in the same way as using a cost matrix built from the inverse class imbalance." Sampling ⇔ cost-sensitivity — two routes to the same bias.

The cost-sensitive class distribution implied by \(T_{CS}\):

\[ n_1' = n_1\cdot\frac{1-T_{CS}}{T_{CS}}, \qquad n_0' = n_0\cdot\frac{T_{CS}}{1-T_{CS}} \]

Inverse-cost example for 99% / 1% imbalance: \(C(0,1)=99,\ C(1,0)=1 \Rightarrow T_{CS}=\tfrac{1}{100}=0.01\), so \(n_1' = 1\cdot\tfrac{0.99}{0.01}=99\) — rebalanced to 99 vs 99.

The three resampling methods

MethodMechanismProCon
Oversamplingduplicate minority rowsno data lostcan overfit duplicates
Undersamplingdrop majority rowsfaster, smallerdiscards data
SMOTEsynthetic minority points between an instance & its \(k\) nearest neighboursricher than copiesneeds some positives; can't invent future cases

SMOTE (Chawla et al., 2002): for each minority point, find \(k\) nearest neighbours, then generate synthetic points on the line between the instance and a neighbour. Variants: Borderline-SMOTE, ADASYN, Safe-level, MWMOTE (see imblearn). Oversampling above 1:1 = using even more extreme costs.

Critical caveats — these are exam traps
  • Resample ONLY the training set. The test set is never touched.
  • After resampling, test accuracy usually DROPS below the "always predict majority" baseline — because the model now flags some positives, creating false positives. That's the price of extracting true positives (precision/recall trade-off). Trap: "resampling improves test accuracy" is generally false.
  • SMOTE is not magic — you still need real positives; it cannot create hidden/future positive cases.
  • Contrary to popular belief, resampling is often NOT worth it with tree-based ensembles or deep learning. Better there: tweak the loss function, conformal techniques. (To SMOTE or not to SMOTE? Elor 2022; Carriero 2025; Hassanat 2022.)

You SMOTE the training data and test accuracy comes out lower than your old "predict-majority" baseline. What's the most likely interpretation?

  1. You accidentally resampled the test set too.
  2. SMOTE failed and should be reverted.
  3. Expected: catching positives introduces false positives, so raw accuracy falls — but the model is now actually useful.

ROC curve & AUC

What the curve plots

The ROC curve (Receiver Operating Characteristic, from 1941 radar operators) plots sensitivity (TPR) on \(y\) vs \(1-\)specificity (FPR) on \(x\), for every cut-off. The perfect model sits in the upper-left corner (TPR 1, FPR 0). The diagonal is the random baseline.

AUC and its probabilistic meaning

The one interpretation to memorize (Hanley & McNeil, 1983)

AUC = the probability that a randomly chosen positive instance gets a higher score than a randomly chosen negative instance. A coin-flip model gives AUC 0.5 (worst); a perfect ranker gives 1.0. AUC is threshold-independent — it summarizes all operating points at once.

Drill the why — why is AUC threshold-independent but accuracy isn't?

AUC only cares about the ranking of scores (does each positive outrank each negative?), not where you draw the line. Accuracy needs a specific threshold to turn scores into labels, so it's one point on the ROC curve; AUC integrates over the whole curve. That's why you can report AUC before committing to a cutoff.

ROC sweep table on the 14-row example (click)

At threshold 0.1: \(tp=9,\ fp=5,\ tn=0,\ fn=0\) → recall 1, precision 0.643, TPR 1, FPR 1 (top-right corner). At threshold 0.78: \(tp=5,\ fp=0,\ tn=5,\ fn=4\) → recall 0.556, precision 1, TPR 0.556, FPR 0 (left edge). Each threshold is one point on the curve.

Drill the why — why does PR beat ROC under imbalance?

With 99% negatives, the confusion matrix is flooded with true negatives. FPR \(= FP/(TN+FP)\) has that huge \(TN\) in the denominator, so even many false positives produce a tiny FPR — the ROC curve looks deceptively great. Precision \(= TP/(TP+FP)\) has no \(TN\) in it, so it stays honest about how polluted your positive predictions are. PR reflects the pain you'd actually feel.

AUC can also mislead when ROC curves cross: one classifier can have 5% higher overall AUC yet be worse in the region you care about (e.g. FPR \(\le 0.001\)). Visual inspection of the curve shape is required — a single AUC number hides it.

CAP, lift, and the threshold-free family

CAP / Lorenz / Power curve & Accuracy Ratio

Sort the population high → low score; plot the cumulative % of positives captured against the % of population reached. The Accuracy Ratio:

accuracy ratio\[ AR = \frac{A}{A+B} \qquad\text{and}\qquad AR = 2\,AUC - 1 \]

where \(A\) = area between actual and random CAP, \(B\) = area between ideal and actual CAP. \(AR = 2\,AUC-1\) is the Gini coefficient. (CAP = Lorenz = Power curve = Captured Event Plot.)

Lift

Lift = how much better your model concentrates positives in the top \(n\%\) than random. If your top 100 contain 60 positives and a random top 100 contain 10, lift = 60/10 = 6. Lift of 1 = random sorting. Closely related: Recall@n / Precision@n for the top-ranked observations — crucial when you deliver a shortlist (recommender systems).

Separation measures: KS, Mahalanobis, divergence, h-measure

KS distance
The max vertical gap between the two cumulative score distributions: \(KS = \max_s \big| P(s\mid y{=}1) - P(s\mid y{=}0)\big|\).
Mahalanobis distance \(M\)
\(M = \dfrac{\lvert \mu_p - \mu_n\rvert}{\sigma}\) — distance between mean scores normalized by the (pooled) spread. Better than Euclidean because it accounts for the standard deviation of the scores.
Divergence \(D\)
\(D = \dfrac{(\mu_p - \mu_n)^2}{\tfrac{1}{2}(\sigma_p^2 + \sigma_n^2)}\).
h-measure (Hand, 2009)
A cost-coherent alternative to AUC. AUC's flaw: it is incoherent in misclassification costs — it implicitly uses different cost distributions for different classifiers, i.e. like grading different models with different rulers, and it can mislead when ROC curves cross.
Threshold-free vs threshold-dependent — the clean split

Depend on the threshold: accuracy, precision, recall, F1, balanced accuracy.
Threshold-independent: AUC, PR-AUC, CAP/AR, h-measure, KS, Mahalanobis, divergence. This is a classic true/false trap.

Automation: one threshold vs two

Plot the score distributions of True (peaks high) and False (peaks low). One threshold set high (~0.8) → precision focus; at 0.5 → accuracy/balanced middle ground; low (~0.25) → recall focus. With two thresholds \(t_1, t_2\): accept-immediately / manual-inspection / reject-immediately bands; the middle goes to humans.

degree of automation\[ \text{automation} = 1 - \operatorname{dist}(t_1, t_2) \]

As the gap shrinks (more automation), fewer cases go to manual review → % correct decreases. You trade accuracy for autonomy.

Regression performance

\(r^2\) and adjusted \(r^2\)

\(r^2\) = the proportion of variation in \(y\) explained by the model:

coefficient of determination\[ r^2 = 1 - \frac{SSE}{S_{yy}}, \quad S_{yy} = \sum_i (y_i - \bar y)^2,\quad SSE = \sum_i (y_i - \hat y_i)^2 \]

Problem: plain \(r^2\) only ever goes up as you add features, even useless ones. The fix is adjusted \(r^2\), which penalizes the number of predictors \(k\):

adjusted \(r^2\)\[ r^2_a = 1 - (1 - r^2)\cdot\frac{n-1}{n-k} \]

It increases only if a new term helps more than chance would predict, and decreases otherwise. Most software reports this even when it's labelled "\(r^2\)".

Trap

"Plain \(r^2\) decreases when you add useless features" → FALSE. Plain \(r^2\) keeps rising; only adjusted \(r^2\) can fall.

AIC, BIC, and error measures

AIC (Akaike)
\(AIC = \dfrac{SSE}{n}\cdot\dfrac{n+k}{n-k}\) — a relative trade-off between goodness of fit and complexity (information lost when the model represents the true process).
BIC (Schwarz criterion)
Closely related to AIC; penalizes complexity too. Lower is better for both.
error measures\[ MSE = \frac{1}{n}\sum_i (y_i-\hat y_i)^2,\quad MAD = \frac{1}{n}\sum_i \lvert y_i-\hat y_i\rvert,\quad RMSE = \sqrt{MSE} \]

RMSE equals the standard deviation of errors for an unbiased model. MAD uses absolute errors (robust to outliers); MSE/RMSE use squared errors (penalize big misses harder).

REC curve & the extrapolation trap

The REC curve is the regression analogue of ROC: plot error tolerance on \(x\) vs % of points predicted within that tolerance on \(y\) (cumulative accuracy). The curve estimates the CDF of the error. The \(x\)-axis error can be squared error, absolute deviation, etc.

Drill the why — predicting the future ≠ extrapolating from training data

The Olympic 100m example: fit lines to men's and women's winning times and they cross far in the future, implying women will eventually "outrun causality." That's nonsense extrapolation — pushing a model far outside the range of its training data. Training on 1900–2000 and forecasting 2200 is a fundamentally different (and unjustified) problem from interpolating within the observed range. Use the appropriate technique; this applies to all model types.

7

You add 5 noise features to a linear regression. Plain \(r^2\) rises from 0.61 to 0.64; adjusted \(r^2\) drops from 0.60 to 0.58. What does this tell you?

  1. The noise features genuinely improved the model, since \(r^2\) rose.
  2. The features are not pulling their weight: plain \(r^2\) always rises, but the adjusted-\(r^2\) drop flags them as noise.
  3. Something is broken — adjusted \(r^2\) can never be lower than plain \(r^2\).

Cross-validation & tuning

The train / validation / test pipeline

Basic pipeline: data → train + test; build on train, do the final evaluation once on test. Add a validation set when the model-building loop needs feedback (e.g. early-stopping trees). The general loop tunes parameters against validation, then builds a final model and touches the test set once. But a single validation split risks lucky hits — that's what CV fixes.

k-fold cross-validation

Split the training data into \(n\) folds. Train on \(n-1\), validate on the held-out fold, rotate so each fold is held out once. You get one evaluation per fold (e.g. 0.75, 0.71, 0.84).

Drill the why — compare configurations by the AVERAGE, not the best fold

Example: \(\lambda{=}0.01\) gives folds 0.90 / 0.70 / 0.70 (avg 0.77); \(\lambda{=}0.02\) gives 0.80 / 0.81 / 0.79 (avg 0.80). Pick \(\lambda{=}0.02\). If you chose by the best fold, \(\lambda{=}0.01\)'s lucky 0.90 would win — but that 0.90 is exactly the kind of fluke CV exists to neutralize. The average is the robust signal; the best fold is noise.

To produce the final model: either select the model from the best fold, or — more commonly — retrain once on the full training data using the winning configuration.

Variants

  • Nested / repeated CV — CV inside CV (outer loop evaluates, inner loop tunes).
  • LOOCV — leave-one-out; each fold holds out a single instance (extreme \(k=n\)).
  • Time-series CV — expanding training window, test point just after train, slide forward. You cannot shuffle time.

Bias, variance, and properties of CV

CV is for tuning; the test set is for true evaluation

Don't conflate them. And: CV gives a pessimistically biased estimate — each fold trains on less than the full data, and most models improve with more training data, so folds slightly underperform the final full-data model. (Trap: "CV is optimistically biased" → FALSE.) LOOCV is approximately unbiased because each fold differs from the full set by only one point. Also: fit preprocessing per-fold, on the train folds only, to avoid leakage.

The bias–variance trade-off (the U-curve):

total error\[ \text{Error} = \text{Bias}^2 + \text{Variance} + \text{irreducible} \]
8

Why does k-fold CV tend to underestimate the final model's performance?

  1. Because it leaks test information, inflating then deflating the score.
  2. Each fold trains on less than the full data, and models usually improve with more data, so fold scores are pessimistically biased.
  3. Because averaging always lowers the maximum achievable score.

Multiclass & multilabel evaluation

Multiclass

The confusion matrix generalizes to \(k\times k\). ROC is harder — multiple "positive" classes give multiple one-vs-rest ROC curves, combined via micro-average (pool all decisions) or macro-average (average per-class). If your technique is binary-only, transform:

One-vs-all (one-vs-rest)One-vs-one
# classifiers (\(k\) classes)\(k\)\(k(k-1)/2\)
Decision rulehighest posterior probabilitymajority voting
Contrastclass vs all otherseach pair

Example one-vs-all: A-vs-notA \(p{=}0.92\), B \(0.18\), C \(0.12\) → predict A. Example one-vs-one: A-vs-B→A, A-vs-C→A, B-vs-C→C → A wins the vote.

Multilabel

An instance can carry multiple labels at once. Metrics adapt: Jaccard index, Hamming loss, multilabel precision/recall. Transforms when unsupported:

Binary relevance

One independent binary classifier per label (label yes/no). Predict every label whose classifier fires. Not the same as one-vs-one/one-vs-all. Simple, but ignores label relationships.

Label powerset

Turn it into a multiclass problem over the powerset of labels: e.g. \([1\,0\,1]\) = Dog & Duck present, Cat absent. Simple, but explodes the number of classes.

Validation is hard: leakage & over-reuse

"If scores are too good to be true, they most likely are" → target LEAKAGE

Feature engineering and transformations must be learned on the training set only. And beware hidden overtraining from over-reuse: even if you save the test set for the very end, running too many "just one more small tuning" loops on the same train/validation split secretly fits the model to that split — and the final test result suddenly disappoints.

A test set is the only guarantee of generalization — not using one is out of the question. If data is genuinely scarce, two risky fallbacks:

  • Use CV directly as evaluation — but limit the number of model types tried, and do feature selection.
  • Fit only simple linear parametric models and report AIC/BIC/\(r^2\) as "degree of fit" — but you have NOT performed a real test. The apparent "lack of fit" comes only from the limited model class.
Drill the why — more features than instances = trivial perfect fit

With \(p \gt n\), the system is under-determined: you can fit any target perfectly, in multiple ways (think solving fewer equations than unknowns). So a great \(r^2\) proves nothing — it's the elephant-fitting problem again. This is the danger zone, and it's why a held-out test (not goodness-of-fit) is the real arbiter.

Unsupervised & monitoring

Unsupervised evaluation is more subjective (what is a "good" clustering?). But if you use an unsupervised method to generate features, test-set discipline returns — the feature generator must be fit on train only.

At deployment, monitor for drift and set warnings that trigger retraining. Concept drift = changes in the statistical properties of data/target that erode model fidelity. Sculley's framing: ML is the "high-interest credit card of technical debt" — deploy it only when you must. Tools: Evidently AI, NannyML, MLflow, ZenML, Kubeflow.

Interpretability: white box vs black box

White-box models (linear/logistic regression, decision trees, rules, RuleFit, Bayesian) are interpretable by construction — the easiest path. Black-box models (tree ensembles, deep learning) need techniques to "open up", answering four questions: explain the model as a whole, which features matter, how features drive predictions, or why a single instance was predicted.

The interpretability–performance trade-off

High interpretability / lower accuracy at the top-left (linear/logistic regression, Bayesian rule lists, decision trees, k-NN); low interpretability / high accuracy at the bottom-right (random forests, SVM, XGBoost, deep learning). The interpretability techniques below buy back some understanding of the bottom-right models. Most are model-agnostic (a few, like MDI and tree inspection, are model-specific).

MDI's failure — a random column can outrank real features

MDI strongly favours high-cardinality (typically numerical) features over binary/low-category ones, because more split points = more chances to reduce impurity on the training set. In the deck, a column of pure random numbers ranks as more important than bedrooms. When a model overfits, MDI rewards features that aren't predictive on unseen data. Treat MDI with suspicion.

Permutation importance

  1. Start with a trained model, a dataset, and a metric (e.g. AUC).
  2. Shuffle (permute) the values of one feature, breaking its link to the target.
  3. Predict again with the same trained model.
  4. Importance = baseline performance − permuted performance.

It captures interactions (shuffling breaks them), but tells you which features matter, not how much or in which direction. Pitfalls: permuting one feature breaks interactions (permute correlated/interacting features together to study them); correlated features share importance, which can mislead.

Drill the why — train set or test set for permutation importance?

Both have a use. On the test set: aligned with evaluation — training-set importance might tell you a feature matters when the model was merely overfitting it. On the train set: learn how the model actually relied on each feature internally. Different questions, different answers.

PDP & ICE: how features drive predictions

Partial Dependence Plots

Each point on a PDP is the average prediction across all observations for a fixed value of the feature under study — showing how that feature impacts the outcome. Two equivalent constructions:

  • Option 1: keep the target feature as-is, impute median/mode for all others, predict, plot over the feature's values (or a grid).
  • Option 2: for each feature value, generate \(N\) synthetic rows by drawing the other features from randomly sampled instances, predict, and average.

2D PDPs (two features) give contour/surface plots; higher dimensions are infeasible.

PDP's ceteris-paribus trap — absence of evidence ≠ evidence of absence

With \(y \sim x_0 x_1 + x_2 + \text{noise}\), the PDPs for \(x_0, x_1, x_2\) all look like flat increasing lines — the interaction \(x_0 x_1\) does NOT show up, because everything else is held fixed (averaged out). A flat PDP doesn't prove a feature is irrelevant.

ELI-intern · why look at the PDP and the raw data?

Looking only at raw data (bin the feature, plot % yes) can imply false correlations that vanish once interactions are accounted for. Example: the data shows "sales drop for customers aged 30–40," but the PDP says "sales stay constant with age." The disagreement is itself the insight: age alone doesn't explain sales — there's an interaction with another feature. So inspect both: the PDP isolates the model's ceteris-paribus view; the data shows the messy reality. Each catches what the other misses.

ICE plots

ICE = PDP's idea, but one line per original instance instead of a single average. For each instance, vary the feature across a grid → one line each. This reveals heterogeneity that PDP's average hides (e.g. the feature helps half the customers and hurts the other half, averaging to "flat"). Lines are often centered (start at 0); the average ICE line equals the PDP.

10

A PDP for age is perfectly flat, yet the feature is clearly used by the model. What's the most careful conclusion?

  1. age is provably irrelevant and can be dropped.
  2. The averaging may be hiding interactions or opposing per-instance effects; check an ICE plot before concluding.
  3. The PDP must be miscomputed, since used features always have sloped PDPs.

LIME: local surrogate explanations

Restate the problem LIME solves: PDP/importance explain a feature or the whole model — what's left unexplained?
Why one specific instance got its prediction. A doctor doesn't want "age matters on average" — she wants "this patient was flagged high-risk because of these specific values." LIME builds a simple, interpretable model that is only accurate locally, around that one instance, to explain that individual decision.

LIME (Ribeiro et al., 2016) builds a local surrogate for an individual prediction. Model-agnostic; works on tabular, text, and image data. Given a black box \(b\) and instance \(x_i\) with \(b(x_i)=p(y{=}1\mid x_i)=0.78\):

  1. Perturb: create a dataset of perturbed samples around \(x_i\); get the black box's predictions.
  2. Weigh each perturbed point by its proximity to \(x_i\) (exponential kernel, or \(k\) nearest neighbours).
  3. Fit a simple, interpretable model on the weighted dataset.
  4. The simple model = a local decision boundary approximating the black box near \(x_i\).
Drill the why — why LASSO regression as the local model?

Regression because the black box outputs a continuous probability (0.78), so the surrogate predicts a number, not a class. LASSO because its \(L_1\) penalty drives coefficients to zero, keeping the explanation sparse — "your loan was denied because of these 3 features" beats a list of 40. Sparsity is interpretability.

Non-tabular: Text — perturb by randomly removing words (easy with bag-of-words). Images — perturbing single pixels is meaningless, so perturb superpixels (groups of similar connected pixels, found via k-means) by "blanking" them.

LIME's pitfalls — why it's less used now
  • Defining the neighborhood / kernel bandwidth is hard and changes results.
  • Choosing the simple model (and tuning its regularization) is somewhat arbitrary — the main reason LIME fell out of fashion. Its advantage remains: easy to understand, works across tabular/text/images.

Shapley values & SHAP

The game-theory idea

From cooperative game theory (Lloyd Shapley). The Shapley value is the fair payout for each player in a cooperative game; in ML it measures a feature's contribution to a prediction. The trick is marginal contribution averaged over all orderings.

The marginal-contribution intuition (click)

Players A, B, C; \(v\) = value of a coalition. \(v(\{A\})=6\); add B → \(v(\{A,B\})=10\), so B contributes +4; add C → \(v(\{A,B,C\})=12\), so C contributes +2. But add C before B: \(v(\{A,C\})=10\), so C contributes +4, then B +2. The contribution depends on order — so to be fair, average each player's marginal contribution over all orderings.

The exact Shapley value for feature \(j\):

Shapley value\[ \varphi_j = \sum_{S \subseteq \{x_1,\dots,x_p\}\setminus\{x_j\}} \frac{|S|!\,(p-|S|-1)!}{p!}\Big[v(S\cup\{x_j\}) - v(S)\Big] \]

Sum over all subsets \(S\) of the other features; the combinatorial weight averages over orderings; the bracket is "value with \(j\) minus without \(j\)". In ML, \(v(X_S)\) = the model's prediction using only the included features, minus the average prediction.

Monte-Carlo approximation

Too many subsets to compute exactly, so estimate with \(M\) samples:

SHAP estimate\[ \hat\varphi_j = \frac{1}{M}\sum_{m=1}^{M}\Big( \hat f(x^m_{+j}) - \hat f(x^m_{-j}) \Big) \]

Each sample: draw a random instance \(z\); build \(x^m_{+j}\) (keeps \(x\)'s value for feature \(j\)) and \(x^m_{-j}\) (uses \(z\)'s value), with the other features randomly split between \(x\) and \(z\); take the difference in predictions.

Drill the why — the additivity property & the killer trap

Shapley values sum to the output minus the base rate. Example: base 0.1 + (age 65: +0.4) + (sex F: −0.3) + (bp 180: +0.1) + (bmi 40: +0.1) = 0.4 = the model's output. Beautiful, additive, complete.

The crucial trap: a Shapley value is the contribution of a feature to this instance's output — it is NOT "how much the output would change if you removed the feature." Mixing these up is the single most common SHAP misconception.

SHAP plots

  • Force plot (one instance): features push the output above/below the base value — red pushes higher, blue lower.
  • Beeswarm / summary plot: features × SHAP value × feature value (color) across all instances — global importance and direction.
  • SHAP dependence plot: a feature's SHAP value vs its value across instances (≈ a PDP); add a second feature as color to see interactions.

SHAP is the most commonly used instance-explainability technique in industry. It combines the best of importance + partial dependence + instance-level explanation.

Accuracy vs AUC

Accuracy

  • Threshold-dependent
  • Misleading under imbalance (99% by predicting majority)
  • Single operating point

AUC

  • Threshold-independent
  • P(random positive scores above random negative)
  • Aggregates all operating points; coin flip = 0.5

Final ELI5 — the whole chapter in one breath

Anybody can memorize the answer key (the training data). The only honest test is a fresh exam (held-out test set) — and you should make that exam hard. Then pick the right ruler for the job (accuracy lies when one answer is rare; use precision/recall/AUC/PR appropriately), tune carefully without peeking at the final exam (cross-validation), and finally ask the model to explain its reasoning (importance, PDP/ICE, LIME, SHAP). Generalize, measure honestly, and understand why.
