The deflationary thesis: Big Data vs Analytics

Before any tool, fix the framing this whole deck is built on. The lecture is the systems deck — but its loudest message is a warning: do not reach for distributed infrastructure you do not need.

Restate the problem in your own words: what does this chapter claim is the danger, and why does "big data" not automatically mean "analytics"?
Teams reflexively reached for Hadoop/Spark for any "big daunting task," over-engineering setups that a single beefy server would handle. The chapter says big data (managing/storing huge, high-speed, unstructured data) and analytics (analyzing data to get insights) are orthogonal: having huge data does not mean you can do ML on it, and doing analytics does not require large volumes. The danger is paying the enormous complexity tax of distribution when your problem does not call for it. The joke "big data is anything which is crash Excel" is a joke, not a definition.
Drill the why · the COST critique

Why be so skeptical of "scaling"? McSherry et al.'s COST ("Configuration that Outperforms a Single Thread") showed an unoptimized distributed system can post impressive scaling curves while a single well-written thread beats the whole cluster. Scaling well is not the same as being fast. So the honest first question is never "which cluster?" but "do I actually have big data, and is the algorithm even the same here?" — the algorithms are the same everywhere.

Why it matters

This framing is the lens for every later tool. HDFS, MapReduce, and Spark are answers to a problem of scale you may not have. In-memory analytics on a \(512\,\text{GB}\) RAM server is often cheaper than an HDFS cluster. The chapter's punchline — "insights > infra" — is an exam-worthy thesis, not a throwaway.

1

"We have 4 TB of clickstream logs, therefore we need a distributed ML platform." What is the cleanest critique of this reasoning?

  1. It is correct — any dataset above 1 TB legally requires Hadoop.
  2. Volume (big data) and analysis (analytics) are orthogonal — large storage needs do not imply you must run ML in a distributed fashion.
  3. It is correct because clickstream data is unstructured and only Spark can read it.

The road to big data: when do you actually need it?

The deck escalates scale deliberately, so you can stop at the first rung that solves your problem.

The rungs of the ladder

In-memory analytics
Whole dataset fits in RAM (pandas, scikit-learn, tidyverse). Simplest and fastest. Servers with \(512\,\text{GB}\) RAM are cheap — often cheaper than an HDFS cluster.
On-GPU analytics
Massively parallel linear algebra (deep learning). Data may be huge (a million images) but is not necessarily distributed; minibatch = gradient over a sample before updating weights.
On-disk analytics
Data exceeds RAM but you still avoid a full big-data stack — the intermediate step. Dask, vaex, Polars, DuckDB use a DAG-of-computations approach with column-oriented storage and aggressive indexing.
Distributed (real big data)
Only here do you split storage (HDFS) and computation (MapReduce/Spark) across many machines.
ELI-intern · the DAG idea you will keep meeting

Notice the recurring theme: Dask, Polars, DuckDB — and later Spark and Tez and Impala — all reach for the same trick: describe your work as a directed acyclic graph (DAG) of operations, then let an engine optimize and schedule it lazily (in memory, columnar) instead of materializing every intermediate step to disk. MapReduce's rigid "map → write to disk → reduce → write to disk" is exactly what this DAG idea kills. When you later read "Spark replaces MapReduce with a DAG engine," you have already met the concept three rungs earlier.

Why it matters

The chapter is blunt: "Pandas is crashing because I'm trying to work with a 50 GB data set is not really an excuse," and "most organizations that jumped on the Spark bandwagon would have been better off" learning apply, a database, or a columnar on-disk library. The road to big data is a road you should be reluctant to finish.

HDFS — distributed storage (but not computing)

Restate the problem: a single disk cannot hold (or survive holding) a petabyte-scale file. What two jobs must a distributed file system do, and what is the one thing it deliberately does not do?
It must (1) spread one logical file across many commodity machines so total capacity scales, and (2) survive machine failures so data is not lost when a cheap node dies. HDFS does the first by chopping files into blocks placed on separate data nodes, and the second by replicating each block. What it deliberately does not do is compute — it just stores. Reading a massive file naively still ends badly; you need MapReduce/Spark on top to process it in a distributed way.

How HDFS works

  • Name node — the brain. Makes all decisions about block placement and replication. (Metadata, not data.)
  • Data nodes — the muscle. Hold the actual data blocks.
  • Blocks — files are split into blocks, typical size \(64\,\text{MB}\); HDFS tries to place each block on separate data nodes.
  • Replication — each block is copied for fault tolerance; the number of replicas is specified at file-creation time.
  • Access — native Java API + a C wrapper + shell commands. "A massive, distributed C: drive."
Drill the why · why 64 MB blocks, not 4 KB?

Ordinary filesystems use tiny blocks (kilobytes) because they juggle many small files. HDFS is built for few enormous files streamed end-to-end. Huge blocks mean (1) the name node tracks far fewer block records (less metadata pressure on the single brain), and (2) once a worker opens a block it reads a long contiguous run — maximizing throughput over latency. HDFS optimizes for "read a lot, sequentially," which is exactly what batch analytics does.

The trap to never forget

HDFS is storage, not computing. Distributing the file does nothing for you if your code still pulls it down and processes it on one machine. You need a distributed computation framework — that is the next section's whole reason to exist.

2

In HDFS, which component decides how blocks are replicated, and what is the typical block size?

  1. The name node decides replication; typical block size is \(64\,\text{MB}\).
  2. Each data node decides its own replication; typical block size is \(4\,\text{KB}\).
  3. YARN decides replication; typical block size is \(1\,\text{GB}\).

MapReduce — distributed computation over (key, value) pairs

Restate the problem: HDFS spread your file across 100 machines. You want a word count. Why can you not just "loop over the file," and what programming shape lets the work happen where the data already lives?
Looping pulls all blocks to one machine — you lose every benefit of distribution and the data may not fit. You need a model where the same simple function runs in parallel, locally, on each block, and partial results are then combined. That shape is map + reduce: map applies a function to every item independently (embarrassingly parallel), then a shuffle routes all values with the same key to one worker, and reduce aggregates each key's group. Computation moves to the data, not data to the computation.

Functional roots

"Map" and "reduce" predate Hadoop — they are functional-programming primitives:

map: a function over every item, producing a new list\[ [1,2,3,4,5]\;.\!\operatorname{map}(\lambda x.\, x^{2}) = [1,4,9,16,25] \]
reduce: aggregate a list down to a scalar\[ [1,2,3,4,5]\;.\!\operatorname{reduce}(\operatorname{sum}) = 15 \]

The Hadoop MapReduce pipeline

  1. Map. Each input \((\text{key},\text{value})\) pair is mapped to a list of output pairs — zero, one, or more. Runs in parallel across all input pairs.
  2. Shuffle / distribute. Output entries are routed so that all entries with the same key land on the same worker.
  3. Reduce. Each worker applies the reduce function to each key's group, producing one final pair per distinct key. Outputs optionally sorted per key.
Drill the why · why must map be allowed to emit zero, one, or many?

Because the relation between input records and output records is not one-to-one. One line of text produces many word-count pairs (one-to-many). A filter step produces zero or one (drop or keep). Forcing exactly one output per input would make map useless for the most common jobs. The flexibility — list-valued output — is what lets map express splitting, filtering, and expansion all at once.

Running it on Hadoop

  • Tasks are written in Java (Python and other bindings exist; Java is native), packaged as a JAR, launched with hadoop jar myfile.jar ClassToRun [args].
  • Output goes to HDFS as a _SUCCESS marker plus part-r-00000 files; inspect with hadoop fs -ls / hadoop fs -cat.
  • Beyond mappers/reducers, real jobs also use Partitioners, Combiners, Shufflers, Sorters (file → InputFormat → Split → RecordReader → map → Partitioner → sort → reduce → OutputFormat).
The honest caveat

"There's a reason most tutorials don't go much further than counting words." Expressing real analytics as map/reduce is genuinely hard, and big does not mean fast — the disk swapping between steps is brutal. Ask whether your use case really resembles a search engine's.

The reduce-associativity trap (the exam-classic)

This is the single most testable mechanics point in the chapter. Slow down here.

Restate the problem: a reducer may be called more than once on partial results — it might combine \([100,200]\) into one value, and a straggler mapper later hands it another value for the same key. What two properties must reduce therefore satisfy?
(1) The reduce function must emit the same structure that map emits, so its output can be fed back in and reduced again. (2) It must give the correct result even when applied multiple times on partial results (it must be associative over partials). We want this because a complete group may be too large to reduce at once, we want to start reducing before all mappers finish, and we keep reducing until one item per key remains.

Why naive averaging is WRONG

Suppose reduce(key, values) = (key, sum(values)/length(values)). For the genre drama:

first reduce of [100, 200]\[ \frac{100+200}{2} = 150 \]

Now a slow mapper emits drama → 300. The engine re-reduces \([150, 300]\):

re-reduce — uh-oh\[ \frac{150+300}{2} = 225 \quad\neq\quad \frac{100+200+300}{3} = 200 \]
Why exactly does it break?

The mean is not associative: averaging the average of a subgroup with a new value silently forgets how many records the average represented. The \(150\) stood for two books, but the re-reduce treats it as one. Information was lost the moment we collapsed to a bare average.

The fix: carry the count

Make the value a pair \((\text{running average},\ \text{number of records seen})\). Map yields (genre, (nrPages, 1)). Reduce reconstructs a weighted sum:

def reduce(key, values):
    sum, newcount = 0, 0
    for (nrPages, count) in values:
        sum = sum + nrPages * count
        newcount = newcount + count
    yield (key, (sum / newcount, newcount))

Now trace it. First, drama → [(100,1),(200,1)]:

weighted, with count carried\[ \left(\frac{100\cdot1 + 200\cdot1}{1+1},\; 2\right) = (150,\,2) \]

Then the straggler arrives with (300,1); re-reduce \([(150,2),(300,1)]\):

re-reduce — correct now\[ \left(\frac{150\cdot2 + 300\cdot1}{2+1},\; 3\right) = (200,\,3) \;\checkmark \]
Drill the why · why does the count make it associative?

Because \((\text{average},\text{count})\) is enough state to reconstruct the true sum (\(\text{average}\times\text{count}\)) at any point. Sums are associative and order-independent: a \(\sum\) of partial sums equals the total sum, always. By carrying the count we never lose the weight, so each re-reduce recomputes the exact running mean regardless of how the data was chunked or how late a mapper finishes. This is the same insight that makes parallel and streaming aggregates correct everywhere.

Why it matters beyond this slide

This is not a Hadoop quirk — it is the general law of distributed/streaming aggregation. Any reducer-like combine step (Spark's reduceByKey, a streaming window aggregate, a Combiner) must be associative and commutative on partial state, or your answer depends on machine timing. "Carry enough state to be re-combinable" is the lesson.

4

Why is naive MapReduce averaging incorrect when a reducer runs on partial results?

  1. Because integer division rounds the result down each time.
  2. Because a bare average loses the count it represented, so re-reducing it with new values weights them wrongly — the mean is not associative.
  3. Because the shuffle step sorts keys, which reorders the values.

YARN and the Hadoop generations

Restate the problem: in Hadoop 1, one service (the JobTracker) both scheduled MapReduce jobs and managed cluster resources. Why was bundling those two jobs a scaling problem?
A single JobTracker became a bottleneck and single point of contention as clusters and task counts grew — it could not scale to larger clusters. Hadoop 2 split the concerns: cluster resource management became YARN (Yet Another Resource Negotiator), while MapReduce kept only the MapReduce-specific logic. Separating "who gets resources" from "what the computation is" lets each scale independently — and lets other engines (like Spark) run on YARN too.

What changed across generations

Hadoop 1
JobTracker manages everything; does not scale to large clusters/tasks.
Hadoop 2
MapReduce split into two: resource management → YARN; MapReduce-specific capability stays in MapReduce. This is why Spark can run on YARN.
Hadoop 3
Mostly the same, plus erasure coding for cheaper durability, and support for multiple standby NameNodes (Hadoop 2 allowed only one active + one standby).
Drill the why · the erasure-coding storage math

Hadoop 2 replication stores 3 full copies → \(3\times\) disk to survive failures. Hadoop 3 erasure coding adds one parity block per two data blocks, so you store the equivalent of \(1.5\) units per unit of data → only \(1.5\times\) disk, with the same fault tolerance. Why is parity cheaper than copies? A parity block lets you reconstruct a lost block algebraically (like RAID), so you do not need a whole redundant copy of everything.

storage overhead, same durability\[ \text{replication: } 3\times \qquad\longrightarrow\qquad \text{erasure coding: } 1.5\times \]
Why it matters

YARN is the reason "the cluster" is decoupled from "the engine." When the chapter later says Spark can run on YARN, Mesos, or standalone, this separation is the precondition. Standard Hadoop, though, is "not a turn-key solution" — just a big distributed drive plus an un-fun way to do MapReduce. That gap is exactly what vendors and Spark rushed to fill.

6

What problem did YARN (Hadoop 2) solve relative to Hadoop 1?

  1. It split cluster resource management out of the JobTracker, which had become a scaling bottleneck.
  2. It replaced HDFS with a faster in-memory file system.
  3. It removed the need for replication by introducing erasure coding.

SQL on Hadoop — and why "big does not mean fast"

Every time a distributed store gets popular, the next demand is: "What do you mean it is a file system? We need SQL!"

Apache Hive (2008)
The original SQL-on-Hadoop. Converts SQL into a series of MapReduce jobs; presents like a MySQL server (CLI, JDBC). Its main drawback is speed — even simple queries pay the MapReduce-translation overhead.
HBase
The first database on Hadoop. No SQL — its own get/put/filter ops (scan, get); column-family + timestamped cells; complex queries become MapReduce jobs.
Pig
A dataflow language to ease writing MapReduce — still not easy, and people still wanted SQL.
Without MapReduce
Tez (a DAG backend for Hive) and Cloudera's Impala (2012) forewent MapReduce entirely for speed. The same DAG idea again.
Drill the why · why is Hive slow even on small queries?

Because Hive does not run SQL — it compiles SQL into MapReduce jobs, and every MapReduce job pays fixed costs: JVM startup, scheduling, and writing intermediate results to HDFS between map and reduce. A trivial GROUP BY still spins up the whole machinery and swaps to disk. The size of the data barely matters to that fixed overhead — hence "big does not mean fast." This is precisely the pain that motivates Spark and the DAG-based engines.

Why it matters · the hype curve

Interest in "big data" peaked between 2011–2014; by 2015 attention moved to AI/ML/deep learning, and 2015 was "the year of Apache Spark." Today storage/querying has stabilized around cloud warehouses (Redshift, Snowflake, Clickhouse, Presto, Dremio) — "do not do it yourself." The analytics story, though, largely returned to BI-style SQL; MapReduce was never well suited for analytics (ask the Mahout project, who tried to port ML to it).

7

What is Hive's main drawback, and why?

  1. It cannot read HDFS files, so it must copy data to a local disk first.
  2. It only supports the get/put/filter API, not real SQL.
  3. Speed — it translates SQL into MapReduce jobs, so even simple queries inherit MapReduce's overhead and disk swapping.

Spark — kicking out MapReduce

Restate the problem: MapReduce is slow because it writes intermediate results to HDFS between every map and reduce, and is painful for iterative jobs (like ML, which loops over the data many times). What single architectural change would fix both?
Keep intermediate data in memory across steps instead of swapping to disk, and let a job be an arbitrary DAG of operations (supporting cyclic/iterative data flow) rather than a rigid map-then-reduce pair. That is exactly what Spark does: it keeps the resource manager and storage (runs on YARN/Mesos/standalone, reads HDFS/S3/Cassandra/etc.) but throws out the MapReduce paradigm, replacing it with a DAG execution engine + in-memory computing. Result: \(10\text{–}100\times\) faster, and finally good for ML and interactive queries.

What Spark keeps vs throws out

Keeps

  • Resource manager: runs on YARN, or Mesos, or standalone.
  • Storage: HDFS, S3, Cassandra, HBase, Hive, flat files (JSON/Parquet/CSV).

Throws out

  • MapReduce. Replaced by an advanced DAG execution engine with cyclic data flow and in-memory computing.
  • Does batch and streaming, interactive queries, ML.
Drill the why · why is in-memory + DAG so much faster for ML?

Iterative algorithms (gradient descent, k-means) sweep the same dataset dozens of times. Under MapReduce, each sweep re-reads and re-writes HDFS — disk I/O dominates. Spark caches the working set in RAM, so iteration \(n+1\) reuses iteration \(n\)'s data with no disk round-trip. The DAG also lets Spark fuse and reorder operations and avoid materializing throwaway intermediates. Benchmarks on the slides: Hadoop ~\(110\,\text{s}\) vs Spark ~\(0.9\,\text{s}\).

Spark's building blocks

A Spark Core Engine at the base; on top: Spark SQL, Spark Streaming, MLlib (machine learning), GraphX (graphs), SparkR — with BlinkDB (approximate SQL) above Spark SQL.

Why it matters

Spark addresses the exact three things that throttled Hadoop adoption: it is faster, easier to program (Scala/Java/Python/R/SQL + an interactive shell), and lends itself to machine learning. Note the family resemblance to Dask and the on-disk DAG libraries — "a smarter way to distribute jobs over machines."

RDD — Resilient Distributed Dataset

Restate the problem: if you keep data in memory across a cluster for speed, a node can die and take its partition with it. Replicating everything in RAM is wasteful. How can Spark be fault-tolerant without copying data everywhere?
By recording, for each RDD, the lineage — the chain of coarse-grained transformations that built it from its source. If a partition is lost, Spark just recomputes it by replaying those transformations on the source data, instead of fetching a stored replica. Efficiency comes from parallelizing across nodes and minimizing replication; resilience comes from the recipe (lineage), not from copies.

What an RDD is

The RDD is Spark Core's primary data abstraction: an immutable collection of elements (text lines, words, objects, images…) partitioned across the cluster and operated on in parallel. The unpacked name:

Resilient
Fault-tolerant via lineage: the chain \(\text{RDD}_1 \to \dots \to \text{RDD}_n\) is logged and can be replayed to rebuild lost partitions.
Distributed
Partitioned across nodes; Spark handles distribution and task management automatically.
Dataset
A collection of data elements; kept in memory where possible (a huge win for iterative queries).

Transformations vs Actions

Transformations

Produce a new RDD (the original is immutable). Lazily evaluated — nothing runs yet.

map, filter, flatMap, union, distinct, groupByKey, reduceByKey, sortByKey, join

Actions

Produce a value or side-effect and trigger execution of the whole pending DAG.

reduce, count, first, take, collect, saveAsTextFile, countByKey, foreach

Drill the why · why are transformations lazy?

If Spark executed each transformation immediately, it could not optimize. By deferring until an action demands a result, Spark sees the entire DAG at once and can fuse maps, push down filters, pick join strategies, and avoid materializing intermediates you never use. Laziness is what turns "a chain of map/filter calls" into "one optimized physical plan." (It also means a bug in a pure transformation chain may not surface until you call an action.)

Writing Spark: SparkContext, PySpark, where work runs

The driver and the context

  • A Spark app can be a JAR (Scala/Java) or driven interactively via the Spark shell; APIs exist in Scala, Java, Python, R, SQL.
  • In PySpark, the driver program runs on the client and sets up a SparkContext (sc) — the connection to the cluster.
  • sc.textFile("README.md") returns an RDD where each element is a line; .count() gives 126, .filter(lambda l: "Spark" in l).count() gives 15.
The most important sentence about PySpark

Data operations are NOT executed on your machine. The client sends them to the cluster; no data is loaded into the client — unless you call .toPandas() (which pulls results back, and can blow up your driver's memory if the result is large).

ELI-intern · what SparkContext really is

Think of sc as a remote control, not a database. Your Python script holds the remote; the cluster holds the data. Every .map/.filter you write is a button press queued on the remote — nothing happens on your laptop. Only an action (or .toPandas()) actually makes the cluster run and, in the .toPandas() case, ship the result back. The context itself bundles the RDD graph, DAGScheduler, TaskScheduler, scheduler backend, listener bus, and block manager.

RDD → DataFrame → Dataset: imposing structure

Restate the problem: RDDs are powerful but "still feel like MapReduce" — low-level map/reduceByKey over free-form elements, with no schema for the optimizer to exploit. What does a data scientist working with tabular data actually want instead?
They want a DataFrame: named, typed columns (a schema), a high-level domain-specific API like pandas, and SQL. Imposing structure also lets a query optimizer (Catalyst) reason about and speed up the computation. So Spark layers DataFrames (and typed Datasets) on top of RDDs — keeping RDDs available for low-level/unstructured/functional work, but making the structured path the default since Spark 2.0.

The three abstractions

RDD
Free-form distributed collection of elements. No schema. Best for low-level control, unstructured data, functional manipulation.
DataFrame
Immutable distributed collection where every element is a set of values in named columns (it imposes structure on top of RDDs). Like a pandas/R data frame. = Dataset[Row] — untyped (runtime-checked).
Dataset[T]
An extension of DataFrame with a type-safe, object-oriented interface: strongly-typed objects mapped to a schema, giving compile-time error checking. Scala/Java only.
Drill the why · why do Python and R only get DataFrames?

A Dataset[T]'s headline benefit is compile-time type safety — catching field/type errors before the program runs, via a Scala/Java case class. Python and R are dynamically typed: they have no compile step at which such checks could happen. So for them the typed API would be meaningless — they get only the untyped DataFrame = Dataset[Row]. Same engine, same Catalyst optimizer; they just lose the static guarantee they could never have enforced anyway.

Why it matters · Spark 2.0 unification

Since Spark 2.0, the DataFrame and Dataset APIs merged: a DataFrame is literally a Dataset[Row]. Spark SQL (built on Spark Core, powered by the Catalyst optimizer) became the primary, feature-rich interface — hiding raw RDDs behind these higher-level abstractions. Use SparkSession (not SparkContext) as your entry point. For R/Python users: keep using DataFrames, knowing they ride on Dataset[Row] underneath.

Why is the strongly-typed Dataset[T] API unavailable in Python and R?

  1. Because Python and R cannot connect to a SparkSession.
  2. Because the Catalyst optimizer only works on RDDs in those languages.
  3. Because they are dynamically typed — there is no compile step at which compile-time type safety could be enforced.

MLlib — scikit-learn on Spark

MLlib is Spark's machine-learning library; its goal is to make practical ML scalable and easy — "scikit-learn on Spark." It provides algorithms (classification, regression, clustering, collaborative filtering), featurization, pipelines, persistence, and utilities.

Drill the why · which package, and why the switch?

There are two APIs. The old spark.mllib was RDD-based ("not a very helpful way of working"). The new spark.ml is DataFrame-based and is the primary API since Spark 2.0. Why move to DataFrames? They give a schema-aware, pipeline-friendly, optimizer-backed surface that mirrors what data scientists already know — exactly the structured abstraction the previous section motivated. Feature parity was reached in Spark 2.3; in Spark 3 the RDD-based API is in maintenance mode (supported, no new features).

13

Which is the primary MLlib API today, and what is the status of the other?

  1. spark.mllib (RDD-based) is primary; spark.ml was removed in Spark 3.
  2. spark.ml (DataFrame-based) is primary; spark.mllib (RDD-based) is in maintenance mode in Spark 3.
  3. Both are deprecated; MLlib was replaced entirely by H2O.

Streaming analytics — three aspects you must not conflate

Restate the problem: "streaming" sounds like one thing, but the chapter insists it is three. Name the three aspects, and say which one is genuinely hard.
(1) Streaming data — plumbing: a source emits events; you store them, access history, ship them around. (2) Streaming training — training an algorithm on a continuous stream (true online learning). (3) Streaming prediction — applying a model to a stream of instances (a deployment matter). The genuinely hard one is streaming training: online algorithms are difficult and need ad-hoc changes per technique. The chapter's punchline: you usually do not need it — store the stream, predict online, but train offline.
Drill the why · when do you actually need online training?

Almost never — even Netflix (\(\sim 9\) million events/second at peak) retrains the recommender matrix in real time only partially; the main training stays offline. The decisive question is how fast the feature space changes, not how fast the data arrives. If concept drift is slow, a monthly/weekly/daily offline retrain plus online prediction is plenty. Online algorithms (streaming regression, online k-means, incremental matrix factorization) are also just hard to find and maintain. Conflating "fast data" with "must learn online" is the classic mistake.

Bounded vs unbounded; engine batch vs streaming

  • Data can be bounded (finite) or unbounded (infinite); the engine can be batch or streaming — and you can mix (treat finite data as a stream; handle infinite data in batches).
  • Finite data: accurate processing is easy — on failure, reprocess; use checkpoints for efficiency. (This is the "resilient" in RDD.)
  • Infinite data: harder — you must reason about time (event creation / ingestion / processing) and aggregate over a window.

Window types

Sliding window

Overlapping windows that slide along the stream by an interval. Size can be count- or time-based.

Hopping (tumbling) window

Non-overlapping, consecutive windows. Also count- or time-based.

Your data arrives fast. According to the chapter, what most determines whether you need online (streaming) training?

  1. The number of events per second arriving from the source.
  2. How fast the feature space changes (concept drift) — not the arrival rate.
  3. Whether the data is stored in Parquet or AVRO.

The streaming pipeline: collectors, brokers, processors

A canonical streaming architecture has five stages — and the chapter is candid that most of them are plumbing, not analytics.

Event collectors (Flume, NiFi)
Gather/centralize events. NiFi builds on Enterprise Integration Patterns. "Not that helpful" for analytics.
Event brokers (Kafka, RabbitMQ)
Message-oriented middleware ("MOM"): validate, transform, route messages — effectively decoupling producers from consumers. Kafka is especially popular and integrates easily with Spark — but "in itself, not much analytics."
Event processors (Spark, Flink, Storm, Samza)
Here is the actual analytics. The two Spark options: Spark Streaming and Spark Structured Streaming.
Drill the why · why insert a broker like Kafka at all?

Decoupling. Without a broker, every producer must know about every consumer, match its pace, and handle its failures — an \(N\times M\) mess. A broker mediates: producers write once, consumers read at their own speed, and the system stays resilient and scalable if either side restarts. It buys you durability and back-pressure handling — but, crucially, it does no learning on its own.

Spark Streaming vs Structured Streaming

Restate the problem: Spark Streaming forces you to think in low-level RDDs and struggles when data arrives late (an event generated at 12:04 but received at 12:11 should still update the 12:00–12:10 window). What cleaner mental model would fix both the API and the late-data problem?
Treat the stream as an unbounded table that is continuously appended, and express your streaming query exactly like a batch query on a static table (DataFrame/SQL). The engine runs it incrementally, maintaining just the minimal intermediate state needed to update results — which lets it correctly update an old window when late data arrives. That is Structured Streaming, built on the Spark SQL engine, replacing the RDD-based DStream model.

Spark Streaming (DStreams)

  • Operates in micro-batching mode — many small batches; often sits on top of Kafka.
  • The abstraction is the DStream (Discretized Stream): a continuous stream represented internally as a sequence of RDDs, each holding one interval's data.
  • Entry point StreamingContext(sc, batchInterval); supports windowed ops, e.g. reduceByKeyAndWindow(f, 30, 10) — window length 30, sliding interval 10.
  • It "heavily uses RDDs (not very pleasant)," but you can convert to DataFrames/SQL via a lazily-instantiated singleton SparkSession.

Structured Streaming

  • A scalable, fault-tolerant engine built on the Spark SQL engine; the key idea is the unbounded table.
  • You write the same code as a batch query; the engine runs it incrementally and continuously, triggered (e.g., every 1s), emitting in complete/append/update mode.
  • Does NOT materialize the whole table — it reads the latest data, updates the result, discards the source, keeping only minimal state.
  • Maintains state long enough to handle late data — an event arriving at 12:11 still correctly increments the 12:00–12:10 window.
Drill the why · how can it "not materialize the table" yet handle late data?

Because it keeps the minimal intermediate state needed to keep each running aggregate correct — not the raw rows. For a windowed count it stores per-(window, key) counters, then discards the source events. When a late event for an old window arrives, the engine still has that window's counter in state and bumps it. It pays memory only for active aggregates, not for the whole infinite history — the same "carry enough state to be re-combinable" lesson from the MapReduce averaging trap.

15

What is the core abstraction of Spark Structured Streaming, and what does it deliberately avoid?

  1. A DStream (sequence of RDDs); it avoids using the Spark SQL engine.
  2. A replicated in-memory copy of the full history; it avoids any state.
  3. An unbounded table queried like static data; it avoids materializing the whole table, keeping only minimal state.

  3. A Kafka topic with no Spark representation.
