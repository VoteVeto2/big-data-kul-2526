The big picture: the pipeline mindset

Before any single technique, internalize the spine of the whole chapter. Supervised learning correlates features to a known target. Your job in preprocessing is to turn raw, messy, multi-table data into a single clean feature matrix — instances as rows, features plus the target as columns — because most tabular techniques assume exactly that single-table input.

Restate the problem in your own words: why is "fit on train, re-apply to test" the single most important rule of this chapter?

Your finished model will run on new, unseen instances that arrive one at a time and cannot look into the future. Any transformation that uses statistics across instances — an imputation median, a WoE value, a scaling \(\mu\) and \(\sigma\), a grouping rule — is itself a learned parameter. If you re-estimate it on the test set, you are (a) leaking information the production model will never have, and (b) producing a number you could not reproduce at scoring time. So you freeze the rule on the training data and re-apply it identically everywhere. Everything you do over multiple instances becomes part of the full model pipeline.

Task types: framing the target

Supervised learning splits into regression (continuous label) and classification (categorical label). Classification has sub-types: binary (most tabular cases — churn, default, fraud, response, maintenance), multiclass, ordinal, and multilabel. Regression has absolute, delta, and quantile flavours.

🧠 Drill the why — why does framing matter?

The lecture stresses "oftentimes, different approaches are possible." The same business question can be posed as regression vs quantile regression, predict-the-value vs predict-the-change, bin-then-classify, reduce-to-two-outcomes, classifier chaining, segment-then-model, or even survival analysis (predict time until event instead of yes/no). The framing decides which leakage risks, which losses, and which preprocessing you face — so it is a modelling decision, not an afterthought.

Constructing the dataset & the leakage trap

Data mining can only find patterns actually present in the data. The dataset must be large enough to contain the pattern, yet concise enough to mine in reasonable time and not so old it swamps the learner with outdated information. Building it is real work: merging sources, choosing aggregation levels (household vs individual), linking IDs, defining the target, cleaning.

⚠ Exam trap — future / leakage variables

Don't include explanatory variables that are too "perfectly" correlated with the target. The detection question: do you know this variable before the target outcome, or only after? At scoring time the model cannot look into the future. Set aside a hold-out test set as soon as possible.

Exploring & cleaning the data

EDA and Anscombe's Quartet

Exploratory Data Analysis means looking at the data: boxplots, scatter plots, histograms, correlation plots (Python: pandas-profiling, AutoViz, SweetVIZ). The canonical lesson is Anscombe's Quartet.

Four datasets share mean \(X=9\), variance \(X=11\), mean \(Y=7.5\), correlation \(r=0.816\), and the same fitted line \(y=3.00+0.500x\). What is the lesson, and why is it dangerous?

Despite identical summary statistics, the four point clouds look completely different (one linear, one curved, one with a single outlier, one with a leverage point). Summary statistics alone are misleading — always visualize. The danger: a pipeline that only checks means/correlations would treat all four as equivalent and miss the structure that determines whether a linear model is even appropriate.

Data cleaning

Three jobs: enforce consistency (K.U.L., KU Leuven, KULeuven are one entity); transform representations to a common format (Male/Man/MAN → M; True/YES/Ok → 1); and remove future variables (a second leakage source — variables that got modified according to the target).

Missing values

Restate the problem: why are missing values a problem at all, and what is the first distinction you should draw?

Many techniques simply cannot consume a missing value (some can in theory but not in their implementation). The first distinction is not-applicable (credit-card limit is NA because the customer has no card — the NA is meaningful) versus unknown / not disclosed (age is NA because nobody recorded it). A secondary axis: missing (completely) at random (MAR/MCAR) vs missing not at random (MNAR), where the missingness itself carries signal.

The three treatment branches

Delete
Drop the whole row or column. Simple, but throws away data.
Replace (impute)
Fill with the mean, the median (more robust), or the mode (for categoricals); or predict the missing value with a separate model — often not worth it in practice.
Keep
Add a separate missing-value indicator feature (a binary flag) so the model can use the fact that it was missing.

The common practical recipe: delete rows with too many missings; impute numeric features with the median and categoricals with the mode; add an indicator column. Sometimes impute within the same class label. More advanced: nearest-neighbor imputation, then missForest / Autoimpute.

🧠 Drill the why — median over mean?

The median is robust to outliers: a single income of 3 million drags the mean upward but barely moves the median. Imputing the mean would inject that distortion into every gap. For categoricals there is no mean, so you use the mode (most frequent level).

⚠ Heavily examinable — the production intermezzo

New unseen instances can have missing values too. Don't impute with a new median! Use the same rules derived from the training data, and ideally apply those rules to validation data too. Never seen a missing value for this feature before? Go back to the original training data to construct the rule, and consider rebuilding the model (monitoring!). Everything you do over multiple instances becomes part of the full model pipeline.

Outliers & duplicates

An outlier is an extreme observation (age = 241, income = 3 million) — and it can be valid or invalid. Some techniques are sensitive to them; sometimes detecting them is the whole task (anomaly / fraud analytics). Detect with histograms, z-scores, box-plots, violin plots.

🧠 Drill the why — valid vs invalid decides the treatment

An invalid outlier (age 241 is impossible) is treated as a missing value. A valid outlier (a genuine millionaire) is real signal — what you do depends on the downstream technique: keep as-is, truncate/cap (often at \(\mu \pm 3\sigma\), pulling the tails inward), or categorize (bin it). You never blindly delete a valid extreme.

Capping bound (truncation)\[ x_{\text{capped}} = \min\!\big(\max(x,\ \mu - 3\sigma),\ \mu + 3\sigma\big) \]

Detection algorithms (sklearn): Robust covariance, One-Class SVM, Isolation Forest (slower, ~0.14s), Local Outlier Factor — each draws different decision contours. Box-plot reading: box = IQR (Q1–Q3), line = median, whiskers reach the non-outlier range, ○ = outlier. Duplicate rows can also be valid or invalid — treat accordingly.

Scaling transformations

Standardization vs normalization

Standardization

Constrain a feature to roughly \( \mathcal{N}(0,1) \).

z-score\[ x_{\text{new}} = \frac{x - \mu}{\sigma} \]

Zero mean, unit variance, unbounded. Good/necessary for Gaussian data and any technique using Euclidean distance or similarity.

Normalization (feature scaling)

Rescale to \([0,1]\) or \([-1,1]\).

min-max\[ x_{\text{new}} = \frac{x - x_{\min}}{x_{\max} - x_{\min}} \]

Bounded fixed range, but sensitive to outliers (the min/max set the scale).

🧑‍💻 ELI-intern — why scale at all, and why never for trees

Imagine k-NN with income (0–100,000) and age (0–100). Euclidean distance is dominated almost entirely by income because its numbers are thousands of times bigger — age is invisible. Scaling puts every feature on a comparable footing so distance is fair. A decision tree, by contrast, only ever asks "is feature ≤ threshold?" — a monotonic rescaling doesn't change the order of values, so the same split exists at a rescaled threshold. Scaling is therefore useless for trees (and harmless), but essential for SVM, regression, k-NN, neural nets.

Categorization: binning & grouping

Categorization (a.k.a. coarse classification, classing, binning, grouping) goes continuous → nominal (binning ranges) or nominal → reduced nominal (grouping levels when there are too many).

Equal-width / interval binning

Equal value ranges on the x-axis → uneven counts (some bins nearly empty).

Equal-frequency binning

Equal counts per bin (a.k.a. histogram equalization) → uneven, less interpretable widths.

🧠 Drill the why — why categorize a perfectly good number?

Four reasons: (1) to treat outliers; (2) to make the model more interpretable; (3) to reduce the curse of dimensionality from too many levels; (4) — the subtle one — to introduce non-linear effects into a linear model. If default-risk-vs-age is non-monotonic (high in the late 20s, then declining), a linear model cannot bend. Binning age lets each bin get its own coefficient, so the line can "step" through the curve. Both equal-width and equal-frequency ignore the target — that is their shared weakness, which odds/WoE binning fixes next.

Encoding categoricals into numbers

"Account activity" is high / medium / low. A colleague encodes it as 1 / 2 / 3 in a single column for a logistic regression. Restate exactly why that is problematic.

Integer encoding imposes a false ordering and equal spacing on categories that have neither. The model (especially linear or distance-based) reads it as a magnitude: it believes low is three times high, and that the gap high→medium equals medium→low. Those numeric relations are fiction, so the fitted coefficient is meaningless.

Dummy (one-hot), binary, and the comparison

Dummy variables (one-hot)
One 0/1 column per level (high/medium/low → 3 columns), then drop one → 2 columns. Mainly used in regression, which can't take categoricals directly.
Binary encoding
Encode the index in binary (1→01, 2→10, 3→11) → ~\(\log_2 k\) columns. More compact, less commonly used.
🧠 Drill the why — why drop one dummy column?

If you keep all \(k\) dummies and an intercept, the columns sum to 1 in every row — they are perfectly linearly dependent (the dummy-variable trap), so the regression's design matrix is rank-deficient and coefficients are not identifiable. Dropping one level makes it the reference baseline; the remaining coefficients are read relative to it. The lecture notes this is "not always needed!" — tree-based and regularized methods don't require it.

High-cardinality categoricals: odds & WoE

A categorical with too many levels (postal codes, NACE codes) can't be dummy-encoded without exploding dimensionality. Approaches: domain grouping, frequency-based grouping (replace the code with how often it appears — e.g. postal_code_count = 23; keep the same rule for validation/production!), odds-based grouping, Weights of Evidence, decision-tree based, and embeddings.

Odds-based grouping

The drawback of equal-width/equal-frequency binning is that they ignore the target. Odds-based grouping fixes this: build a pivot of attribute vs target, compute the odds (good/bad) per level, and group levels with similar odds. A chi-square statistic compares grouping options.

Weights of Evidence (WoE) & Information Value (IV)

WoE for a category\[ \text{WoE}_{\text{cat}} = \ln\!\left( \frac{p_{c1,\text{cat}}}{p_{c2,\text{cat}}} \right) \]

where \(p_{c1,\text{cat}}\) = (class-1 instances in this category) / (all class-1 instances), and similarly for class 2. Sign: if \(p_{c1,\text{cat}} \gt p_{c2,\text{cat}}\) then \(\text{WoE} \gt 0\); otherwise \(\text{WoE} \lt 0\).

🧠 Drill the why — the key property of WoE

WoE has a monotonic relationship with the target. Higher WoE indicates less risk. That monotonicity is exactly what a linear/logistic model wants: replacing a categorical with its single WoE column gives the model a feature that already lines up with the log-odds it is trying to fit — one column, no dummy explosion, and a sensible ordering.

Information Value\[ \text{IV} = \sum_{\text{cat}} \big(p_G - p_B\big)\cdot \text{WoE}_{\text{cat}} \]

IV screens variables: important ones have IV > 0.1. Category boundaries are chosen to maximize IV.

⚠ The crucial pitfall

WoE and IV use information from the target. Therefore they must be computed after the train/test split, on the training fold only — otherwise you leak the test labels into your encoding. This is the same "fit on train" rule as imputation, now with teeth.

Laplace smoothing and the categories trade-off

Smoothed WoE\[ \text{WoE}_{\text{cat}} = \ln\!\left( \frac{p_{c1,\text{cat}} + n}{p_{c2,\text{cat}} + n} \right) \]

Larger \(n\) → less reliance on the data, pushing WoE toward 0 (stabilizing rare categories). The categories trade-off: fewer → simpler, more stable; more → keep predictive power. Practical recipe: a sensitivity analysis — plot IV vs number of categories and pick the elbow. Fewer instances in a category → a less reliable WoE.

Other creative encodings & embeddings

Build a decision tree on the single variable + target to suggest bins; 1-D k-means on a continuous variable; Leave-One-Out Mean (Owen Zhang, Kaggle). For very high cardinality, embeddings (from the textual domain — "word2vec and friends"). Entity Embeddings of Categorical Variables (Guo & Berkhahn, 2016) learn a Euclidean mapping via a neural net during training: less memory than one-hot, similar values land close together, and it defines a usable distance for categoricals — especially good where other methods overfit.

Mathematical transforms & interactions

Logs, square roots, Box-Cox, Yeo-Johnson and other power transforms are mostly done to enforce linearity or more Gaussian behaviour (e.g. to linearize a saturation curve flattening toward 100%). Interaction variables matter when neither \(x_1 \sim y\) nor \(x_2 \sim y\) is linear, but \(f(x_1,x_2) \sim y\) is — usually \(f(x_1,x_2) = x_1 \times x_2\).

Connection — it always depends on the learner

Trees capture interactions and non-linearities automatically; linear models need them explicitly. This "depends on the learner" refrain recurs all chapter — it is why you must know your technique's bias before engineering features.

Feature engineering

Definition

"The aim of feature engineering is to transform dataset variables into features so as to help the analytical models achieve better performance in terms of either predictive performance, interpretability or both."

Take the bias of your technique into account (e.g. craft features to make data linearly separable for a linear model). The coordinate-transform example: two concentric rings are not linearly separable in \((x,y)\) but become separable in polar coordinates \((r,\theta)\) — a feature transform turns a hard problem easy. Most techniques are "dumb"; domain knowledge does the heavy lifting. Remember: correlation ≠ causation, and stay within the feature space when extrapolating.

RFM, time, and date features

RFM (Cullinan, 1977): Recency, Frequency, Monetary — popular in marketing and fraud. Recency is often modelled as exponential decay:

Recency feature\[ \text{recency} = e^{-\gamma t} \]

Larger \(\gamma\) → faster decay; never occurred → 0; quite recent → ≈ 1.

🧑‍💻 ELI-intern — why time needs the von Mises distribution

Time is circular: 00:00 = 24:00, and there is no natural ordering (is 23:00 > 01:00?). The naive arithmetic mean of 23:00 and 01:00 is 12:00 — the exact opposite side of the clock, which is nonsense. So you model time as a periodic variable using the von Mises distribution (a normal wrapped around a circle), with two parameters: \(\mu\) = the periodic mean, and \(\kappa\) a concentration such that \(1/\kappa\) is the periodic variance. The mean vector points to the correct circular average. For dates with a year, don't use the raw date — derive time-since, quarter, month, week, weekend/holiday flags, and sin/cos transforms for smooth cyclicity.

Deltas, trends, ordinals, relational data

Evolutions over time matter. Either keep an instance as separate rows (panel data), or pick a base time and add relative features:

Absolute & relative trend\[ \text{abs} = \frac{F_t - F_{t-x}}{x}, \qquad \text{rel} = \frac{F_t - F_{t-x}}{F_{t-x}} \]

Beware denominators = 0! For ordinal features (credit ratings) don't use integer coding; use percentile coding (map to cumulative percentiles) or thermometer coding (each higher level fills one more flag, preserving order: AAA = 0000, AA = 1000, A = 1100, B = 1110, C = 1111). Relational data needs denormalization — merging normalized tables into one aggregated table (tools: dm in R, skrub in Python).

Featurization & feature selection

Featurization = squeezing a representational vector out of unstructured data (text, images, signals); deep learning is often used here as a feature extractor, not the final model (featuretools, stumpy, tsfresh, Nixtla). Feature selection: with far more instances than features you're fine; otherwise techniques "go haywire". Basic screening: low variance, chi-square, information gain, correlated features, IV. Advanced: stepwise, regularization, genetic algorithms, variable-importance retraining, PCA.

Linear regression — the baseline

Model\[ y = \beta_0 + \beta_1 x_1 + \beta_2 x_2 + \dots + \varepsilon, \qquad \varepsilon \sim \mathcal{N}(0,\sigma) \]

\(\beta_0\) is the mean response at \(\vec{x}=0\); \(\beta_1\) is the change in mean response per unit increase of \(x_1\). The weights are the only parameters we optimize, by minimizing a loss function:

Ordinary Least Squares (OLS)\[ \arg\min_{\beta}\ \sum_{i=1}^{n}\big(y_i - \hat{y}_i\big)^2 \]
🧠 Drill the why — why squared errors?

SSE/MSE is convex (a single minimum) and fully differentiable, so optimization is well-behaved. If \(X\) has full rank there is a unique closed-form solution. Other optimizers exist (BFGS, L-BFGS-B, Nelder-Mead, genetic algorithms) — they differ in what they assume about the function (linear? differentiable? convex?) and the parameters. Gradient descent is a simple workhorse, often preferred over the closed form on large datasets because the closed form is computationally intensive.

Logistic regression

Restate the problem: give three reasons why you can't just use linear regression for a yes/no classification.

(1) The output \(\hat{y}=\beta_0+\beta_1\text{age}+\dots\) gives no guarantee of 0 or 1. (2) Even read as a probability it has no guarantee of lying in [0,1] (it can shoot to 5 or −3). (3) The target and the errors are not normally distributed, violating the core OLS assumption. The fix is to bound the linear predictor through the logistic/sigmoid function.
Logistic / sigmoid\[ f(z) = \frac{1}{1 + e^{-z}} \]
⚠ Exam classic — is it regression or classification?

Logistic regression is a classification technique, not regression — the target is a binary categorical label (0/1, no/yes). Yet its predictions are continuous in the open interval \((0,1)\): a probability. Both halves of that sentence are true and examinable.

Probability & logit forms\[ P(\text{yes}\mid \vec{x}) = \frac{1}{1+e^{-(\beta_0+\beta_1 x_1+\dots)}}, \qquad \ln\!\left(\frac{P(\text{yes})}{P(\text{no})}\right) = \beta_0 + \beta_1 x_1 + \dots \]

Interpreting coefficients: the odds-ratio

When \(X_i\) increases by 1: the logit shifts additively, \(\text{logit}\big|_{X_i+1} = \text{logit}\big|_{X_i} + \beta_i\); equivalently the odds multiply:

Odds-ratio & doubling amount\[ \text{odds}\big|_{X_i+1} = \text{odds}\big|_{X_i}\cdot e^{\beta_i}, \qquad \text{doubling amount} = \frac{\log 2}{\beta_i} \]
🧠 Drill the why — why does +1 in \(X_i\) multiply the odds?

Because the model is linear in the log-odds. Adding \(\beta_i\) to a logarithm is the same as multiplying the underlying odds by \(e^{\beta_i}\) (since \(e^{a+b}=e^a e^b\)). So \(e^{\beta_i}\) is the odds-ratio: the multiplicative change in odds per unit of \(X_i\), other variables held constant. \(\beta_i \gt 0 \Rightarrow e^{\beta_i} \gt 1\) (odds rise); \(\beta_i \lt 0 \Rightarrow e^{\beta_i} \lt 1\) (odds fall). The doubling amount \(\log 2 / \beta_i\) is how much \(X_i\) must change to double the odds.

The loss: cross-entropy

Binary cross-entropy (log loss)\[ \mathcal{L} = -\big(y\log p + (1-y)\log(1-p)\big) \]

As the predicted probability \(p\) approaches the true label, loss → 0; as it goes the wrong way, loss → ∞. Logistic regression's appeal: the model is a formula — easy to build, interpret, and it is well-calibrated with statistical rigor. Its boundary is linear (interactions can be added explicitly). It is sensitive to outliers, correlated features, and the curse of dimensionality, and categoricals must be encoded first.

Stepwise selection & its dangers

Motivation: prefer a parsimonious model — if a smaller model works as well, use it (and dodge the curse of dimensionality). Selection-based approaches build the model step-by-step: forward, backward, or hybrid (R: stepAIC, regsubsets; sklearn's SequentialFeatureSelector is greedy and uses cross-validation).

⚠ Examinable — stepwise is statistically illegitimate (Harrell, 1996)

Stepwise selection yields \(R^2\) biased high, \(F\)/chi-squared tests without their claimed distributions, falsely narrow confidence intervals, p-values without proper meaning, biased coefficients needing shrinkage, severe collinearity problems — and more data barely helps. Data dredging (developing and confirming on the same dataset) is the root sin; even the AIC is itself a random variable. Take-away: use a proper train–validation–test setup.

Regularization: Lasso vs Ridge

Restate the motivation: the worked example shows a line with a steeper slope can have lower SSE. Why does that motivate a penalty on weights?

SSE alone rewards whatever fits the training points best, even if that means large, unstable coefficients chasing a few points (in the lecture, Model 2's SSE = 8 beats Model 1's 16). Large weights make the model sensitive to outliers and generalize poorly. Adding a penalty on the size of the weights keeps them small/constrained, trading a little training fit for much better generalization.
Lasso (L1) and Ridge (L2)\[ \arg\min_{\beta}\ \sum_i (y_i-\hat{y}_i)^2 + \lambda\sum_{j=1}^{p}|\beta_j| \qquad\text{vs}\qquad \arg\min_{\beta}\ \sum_i (y_i-\hat{y}_i)^2 + \lambda\sum_{j=1}^{p}\beta_j^2 \]

No penalization on the intercept! The key insight: constrained, rather than fewer, parameters. This makes the model less sensitive to outliers and improves generalization.

🧑‍💻 ELI-intern — diamond vs circle, and why Lasso zeroes coefficients

Picture the SSE as elliptical contours and the penalty as a budget you can't overspend. The L1 budget region is a diamond with sharp corners on the axes; the L2 region is a circle. The solution is where the ellipse first touches the budget region. A diamond's corners stick out, so the ellipse very often touches at a corner — and a corner means one coefficient is exactly 0. A circle has no corners, so coefficients shrink toward 0 but rarely hit it. That's why Lasso does feature selection "for free" and Ridge only shrinks.

🧠 Drill the why — why must you normalize before regularizing?

The penalty \(\lambda\sum|\beta_j|\) (or \(\sum\beta_j^2\)) treats every coefficient on the same scale. But a coefficient's size depends on its feature's units — income in euros gets a tiny coefficient, age in years a larger one. Without normalization the penalty would hammer some features and ignore others arbitrarily. Normalize first so \(\lambda\) affects all variables similarly. Pick \(\lambda\) by cross-validation.

Elastic net

Elastic net\[ \arg\min_{\beta}\ \sum_i (y_i-\hat{y}_i)^2 + \lambda_1\sum_j|\beta_j| + \lambda_2\sum_j\beta_j^2 \]

Combines L1 and L2: keeps Lasso's sparsity while gaining Ridge's grouping. Needs a grid search over two parameters (libraries disagree on how to name \(\lambda_1, \lambda_2\)).

Beyond the basics (skim-level)

Know these by name and one-line idea: non-parametric regression (LOESS — flexible smoother, needs lots of data, "more exploratory than a model"); GAMs (splines in a linear combination, capture non-linearities, overfitting risk); multinomial logistic (K classes → K−1 binary models); ordinal logistic (the proportional-odds model — parallel logits differing only in intercept); PCR (PCA then regress, ignores target, scaling-sensitive) vs PLS (decomposition including the target, often behaves like PCR); and TLS/ODR (minimize orthogonal distance — assumes both x and y noisy — whereas OLS assumes x exact, y noisy).

Decision trees

Trees work for (multi)classification and regression by recursively partitioning the data. Three core decisions: the splitting, stopping, and assignment decision. Algorithms: ID3, C4.5, J48, CART. The greedy loop: from the root, try all candidate splits, score each, pick the best, recurse on each subset.

Impurity: entropy, Gini, classification error

Why do we even need an "impurity measure"? What is a pure vs impure node?

A pure node holds only one class (all yes or all no) — impurity is minimal. A 50/50 mix is maximally impure. We want splits that produce pure children, but to compare candidate splits quantitatively we need a number. Impurity measures provide that number, so the algorithm can rank candidate splits and pick the best one.
Entropy, Gini, classification error\[ H(S) = -\sum_{x\in X} p(x)\log_2 p(x), \quad \text{Gini}(S) = 1-\sum_x p(x)^2, \quad \text{ClassErr}(S) = 1-\max_x p(x) \]

\(H(S)=0\) means a completely pure set. Information-theoretically, an observation that halves the space of possibilities carries \(-\log_2(1/2)=1\) bit; quartering it carries 2 bits. Entropy is the expected number of bits — but it is only bounded between 0 and 1 for the two-class case.

Information gain of a split\[ \text{IG}(A,S) = H(S) - \sum_{t\in T} p(t)\,H(t), \qquad p(t) = \frac{|t|}{|S|} \]
Gini vs entropy in practice

All three are 0 at \(p=0\) and \(p=1\) and peak at \(p=0.5\): entropy peaks at 1.0, Gini and class error at 0.5 (class error is triangular/piecewise-linear). Gini is the default in most libraries — slightly faster and a little better for continuous variables.

Algorithms: ID3 → C4.5 → C5

ID3 (Quinlan, 1979/1986)
Most basic; categorical features only; grows as long as it can; greedy — never backtracks, so global optimality is not guaranteed; can't handle continuous variables → horribly overfitting trees.
C4.5
Extension of ID3, still uses information gain. Main contribution: continuous variables. Can handle missing values (the paper says ignore them in the IG computation — though most implementations don't), supports attribute weights, and describes pruning.
C5 / See5
Commercial; faster, smaller trees, misclassification costs, boosting support; not widely adopted, somewhat outdated.
🧠 Drill the why — why binary splits, even for categoricals?

Information gain is biased toward splits that produce more subsets (more branches mechanically reduce impurity). Forcing every split to be binary (e.g. temperature ≤ 21? yes/no) removes that bias, lets you re-use an attribute deeper in the tree, and yields trees that are deeper but less wide. For continuous variables, only the distinct values seen in training are tested as thresholds — anything between them gives the same gain.

Preventing overfitting: early stopping vs pruning

You spot overfitting by plotting accuracy vs tree size: training accuracy keeps rising while validation accuracy peaks then declines — the gap is overfitting.

Early stopping (pre-pruning)

Stop growing on a criterion: too few instances in a subset, depth too high, or validation performance starts to drop.

Pruning (post-pruning)

Grow the full tree, then merge back low-value leaves. "Inside every big tree is a small, perfect tree." Forms: weakest-link, cost-complexity (common), governed by a complexity parameter.

🧠 Drill the why — why can a fully-grown tree not give a probability?

If you grow until every leaf is pure, each leaf is 100% one class, so the only "probability" it can output is 0% or 100% — useless for ranking. If you prune or stop early, a leaf might hold "8 A vs 2 B" → \(P(A\mid x)=80\%\). These probabilities let you rank predictions across instances, but they are not well-calibrated and ignore leaf-node frequency by default.

11

Information gain has a built-in bias. What is it, and how do binary splits help?

  1. It favors splits with more subsets; binary splits remove that bias
  2. It favors continuous variables; binary splits balance categorical and continuous
  3. It favors the majority class; binary splits re-balance the classes

Trees: final properties

  • Preprocessing: robust to outliers; only missing-value treatment needed (some implementations do three-way splits yes/no/NA).
  • Interactions & non-linearities: captured by default.
  • Unstable & not well-calibrated: a small change in training data can produce a very different-looking tree.
  • Boundary: axis-aligned rectangles ("squares orthogonal to a dimension").
  • A deep tree isn't necessarily overfit, and a short tree isn't necessarily fine; leaves are rarely 100% pure. Trees are the backbone of strong tabular learning (ensembles come later).

Regression trees

Made popular by CART (Classification And Regression Trees). Instead of voting classes, a leaf predicts the mean of the continuous label. The split criterion minimizes squared residuals — find splits with minimal within-node variance (hence greatest between-node variance, a bit like k-means). Predictions are piecewise-constant step functions; a model tree fits a linear regression in each leaf for a piecewise-linear fit. A regression tree typically needs to be deeper than a classification one.

12

What does the decision boundary of a decision tree look like, compared to logistic regression?

  1. Both draw smooth diagonal boundaries
  2. The tree makes axis-aligned rectangles; logistic regression a single linear boundary
  3. The tree is linear; logistic regression is rectangular

k-Nearest Neighbors

A non-parametric method for classification and regression. "The data is the model" — "there's a model that perfectly fits the training data: it's called a database." For a new instance, find the \(k\) closest training points by distance; classify by majority vote, regress by a (weighted) average of the \(k\) neighbors, weighted by the inverse of their distance. Because it is distance-based, normalization/standardization is required.

🧑‍💻 ELI-intern — what "lazy learner" really means

Most models do work up front (fit coefficients, build a tree) and then predict cheaply. k-NN flips this: it learns nothing at training time — it just stores the data. All the work happens at prediction time, when it scans the training set to find neighbors. Consequences: you get no formula or tree to summarize, you must keep the whole training set around, scoring is relatively slow, and it's unstable for noisy data and sensitive to \(k\). Yet given enough data it gives surprisingly good results, and it's the natural backbone of a recommender system.

13

Why is k-NN called a "lazy learner"?

  1. It gives poor results regardless of the data
  2. It learns no summarizing model — it stores the data and computes only at prediction time
  3. It only works on low-dimensional data
