Supervised Learning — Evaluation & Interpretability

It's all about generalization. How to know a trained model is any good — and how to know why it predicts what it predicts.

Big picture: it's all about generalization

Why this matters

Performance on the data you trained on tells you almost nothing. It is trivial to reach 100% train accuracy — a fully grown decision tree, a k-NN with \(k=1\), or just storing every row in a database all achieve it. So the whole discipline of evaluation exists to answer one question: does the model work on unseen data?

Generalization
How well a model performs on unseen data (future, noisy, robust conditions). The crux of ML.
Development / Train set
Data used to build the model. Do anything you want here.
Test (hold-out) set
Strictly separate data, used once, to measure performance objectively. Make it as challenging as possible.
The central thesis

The goal of the test set is to catch failure cases — NOT to show off great numbers, NOT to make it weak, and NOT to reuse it until the results look good. A weak or over-reused test set defeats the entire purpose.

The golden rule: split FIRST

  • Split into train / test before any preprocessing or featurization — otherwise information leaks from test into train.
  • Use a stratified split: same class distribution in train and test.
  • Prefer out-of-time when possible (see below).
Statisticians' alternative

Restrict to simple, linear, parametric models and judge "goodness of fit" (AIC / BIC / \(r^2\)). This sidesteps generalization by limiting complexity — but it is not a real predictive test. With more features than instances you can fit any target perfectly, so "fit" alone is meaningless (Boué 2019: any dataset fits with a single parameter).

Out-of-sample vs out-of-time

Out-of-sample

Train and test from the same time period, different instances. Test points scattered among train points.

Out-of-time

Test set from a later time period than train. Appealing because "time always progresses while the model is in use." More honest, but you can never foresee everything.

Confusion matrix & threshold metrics

The confusion matrix

Predicted probabilities become labels via a threshold (e.g. score \(\ge 0.50 \Rightarrow\) "yes"). The matrix — and every metric derived from it — depends on that threshold.

The metrics you must memorize

MetricFormulaPlain language
Accuracy\((TP+TN)/\text{total}\)fraction correct overall
Recall = Sensitivity = TPR\(TP/(TP+FN)\)of the actual positives, how many caught?
Precision = PPV\(TP/(TP+FP)\)of the predicted positives, how many right?
Specificity = TNR\(TN/(TN+FP)\)of the actual negatives, how many correctly cleared (not flagged)?
FPR = Fall-out\(FP/(TN+FP)=1-\text{spec}\)negatives wrongly flagged
Balanced accuracy\((\text{recall}+\text{spec})/2\)average of the two rates
F1\(2\cdot\dfrac{\text{prec}\cdot\text{recall}}{\text{prec}+\text{recall}}\)harmonic mean of P and R
Worked: TN=3, FP=2, FN=2, TP=7 (total 14) \[ \text{Acc}=\tfrac{3+7}{14}=0.71,\quad \text{Recall}=\tfrac{7}{9}=0.78,\quad \text{Prec}=\tfrac{7}{9}=0.78 \] \[ \text{Spec}=\tfrac{3}{5}=0.60,\quad \text{BalAcc}=\tfrac{0.78+0.60}{2}=0.69,\quad F1=0.78 \]
⚠️ Exam watch-out

Recall and precision share the numerator \(TP\) but differ in the denominator: recall divides by actual positives \((TP+FN)\); precision by predicted positives \((TP+FP)\). F1 is the harmonic mean \(2PR/(P+R)\), not the arithmetic mean. Balanced accuracy \(=(\text{recall}+\text{specificity})/2\). And \(\text{FPR}=1-\text{specificity}\).

Tuning the threshold

Each threshold \(t\) gives a different confusion matrix. As the threshold rises, precision generally rises (you only flag confident cases) and recall generally falls (you miss borderline positives). Sweep the threshold and plot the metric you actually care about.

Tuning beyond the threshold

Most traditional models optimize a default surrogate loss (log-loss, impurity, squared error) — rarely the metric you actually care about. Levers: tune the threshold, adjust parameters, redefine the target, sample/filter the data, apply misclassification costs, apply instance weighting (easiest: duplicate rows), or change the loss. Evaluate on what matters to you.

Class imbalance, costs & sampling

Never trust accuracy alone

If positives are 1% of cases, a model optimizing accuracy predicts everything negative: TN=99, FN=1, FP=0, TP=0 → 99% accuracy, zero positives caught. Never believe a report that only mentions accuracy.

Misclassification costs & cost-sensitive learning

A cost matrix \(C(\text{predicted},\text{actual})\) penalizes error types differently. "Missing a positive is 5× worse than a false alarm" → \(C(0,1)=5,\ C(1,0)=1\).

Expected loss of classifying x as class j · decision rule \[ l(x,j)=\sum_{k} p(k\mid x)\,C(j,k),\qquad \text{classify positive} \iff l(x,1) \lt l(x,0) \]
Worked: p(1|x)=0.22, C(0,1)=5, C(1,0)=1 \[ l(x,0)=0.78\cdot 0 + 0.22\cdot 5 = 1.10,\qquad l(x,1)=0.78\cdot 1 + 0.22\cdot 0 = 0.78 \] \[ 0.78 \lt 1.10 \;\Rightarrow\; \text{classify positive} \]

Equivalently, set \(l(x,1)=l(x,0)\) to get the cost-sensitive threshold:

Cost-sensitive threshold \[ T_{CS}=\frac{C(1,0)-C(0,0)}{\big(C(1,0)-C(0,0)\big)+\big(C(0,1)-C(1,1)\big)} \]
  • Symmetric unit costs (\(C(1,0)=C(0,1)=1\), diagonal 0) → \(T_{CS}=\tfrac{1}{1+1}=0.5\) (the default!).
  • \(C(0,1)=5,\ C(1,0)=1\) → \(T_{CS}=\tfrac{1}{1+5}=0.167\). Since \(0.167 \le 0.22\) → positive (same decision as the loss calc).
  • Costs from the inverse class distribution: for 99%/1% imbalance, \(C(0,1)=99,\ C(1,0)=1\), so \(T_{CS}=\tfrac{1}{100}=0.01\).

Resampling for imbalance

The key equivalence

Sampling the dataset so the minority class equals the majority class biases the classifier exactly like a cost matrix built from the inverse class imbalance. Sampling ⇔ cost-sensitivity.

MethodMechanismProCon
Oversamplingduplicate minority rowsno data lostcan overfit duplicates
Undersamplingdrop majority rowsfaster, smallerdiscards data
SMOTEsynthetic points between a minority instance and its \(k\) nearest neighborsricher than duplicatesneeds some positives; can't invent future cases
⚠️ Exam watch-out — the imbalance traps
  • Resample ONLY the training set. The test set is never touched.
  • After resampling, test accuracy usually DROPS below the "always-majority" baseline — you now produce false positives as the price for catching true positives. (Trap: "resampling improves test accuracy" → false.)
  • SMOTE oversamples the minority (it does not undersample, though it can be combined with it) and cannot create hidden/future positives.
  • Contrary to popular belief, resampling is often NOT worth it for tree ensembles or deep learning — better to tweak the loss or use conformal techniques.

Threshold-independent metrics: ROC, AUC, PR

ROC curve & AUC

The ROC curve plots sensitivity (TPR) on the y-axis vs \(1-\)specificity (FPR) on the x-axis, sweeping every cut-off. A perfect model reaches the upper-left corner; a coin flip lies on the diagonal.

AUC interpretation (Hanley & McNeil)

AUC = the probability that a randomly chosen positive instance scores higher than a randomly chosen negative one. Random model AUC = 0.5. AUC is threshold-independent.

⚠️ Exam watch-out

For imbalanced data the PR curve beats ROC. Reason: abundant true negatives dominate the matrix, shrinking FPR and making ROC look deceptively good, whereas precision is unaffected by the TN count. AUC can also mislead when ROC curves cross or when you only care about a sub-region (e.g. very low FPR).

Other threshold-independent summaries

CAP / Lorenz / Power curve
Sort population high→low score; plot cumulative % of positives captured vs % of population. Accuracy Ratio \(AR = A/(A+B)\), and crucially \(AR = 2\cdot\text{AUC} - 1\) (the Gini coefficient).
Lift
How much better than random your model concentrates positives in the top \(n\%\). Model finds 60/100 vs random 10/100 → lift = 6. Lift of 1 = random. Recall@n / Precision@n: same idea for shortlists (recommender systems).
h-measure (Hand 2009)
A coherent alternative to AUC. AUC is incoherent on costs: it effectively uses different cost distributions for different classifiers — like measuring rivals with different rulers. Also misleads when ROC curves cross.
KS distance
Separation measure: \(KS = \max_{s}\,\lvert P(s\mid y{=}1) - P(s\mid y{=}0)\rvert\) — the max vertical gap between the two cumulative score distributions.
Mahalanobis & Divergence
\(M = \dfrac{\lvert\mu_p-\mu_n\rvert}{\sigma}\) — better than Euclidean because it accounts for the spread \(\sigma\). Divergence \(D = \dfrac{(\mu_p-\mu_n)^2}{\tfrac{1}{2}(\sigma_p^2+\sigma_n^2)}\).
Automation: one vs two thresholds

One high threshold (~0.8) → favour precision; one low (~0.25) → favour recall; 0.5 → balanced. With two thresholds, accept above \(t_2\), reject below \(t_1\), and send the middle band to humans. Degree of automation \(= 1 - \text{dist}(t_1,t_2)\); as the gap shrinks (more automation), % correct decreases.

Regression performance

\(r^2\) (coefficient of determination)
Proportion of variation in \(y\) explained by the model.
r² and adjusted r² \[ r^2 = 1 - \frac{SSE}{S_{yy}},\quad SSE=\sum_i (y_i-\hat y_i)^2,\ \ S_{yy}=\sum_i (y_i-\bar y)^2 \] \[ r^2_a = 1 - (1-r^2)\cdot\frac{n-1}{n-p-1} \quad (n=\#\text{obs},\ p=\#\text{predictors, excl. intercept}) \]
⚠️ Exam watch-out — r² vs adjusted r²

Plain \(r^2\) only ever increases as you add predictors, even useless ones. Adjusted \(r^2\) can decrease: it rises only if a new term improves the model more than chance. (Trap: "plain \(r^2\) decreases when you add junk features" → false.)

AIC / BIC
Trade off goodness of fit vs complexity. For Gaussian regression \(\text{AIC}=n\ln(\text{SSE}/n)+2k\) (\(k=\#\)parameters). BIC = Schwarz criterion, replaces \(2k\) with \(k\ln n\) (penalizes complexity more).
Error measures
\(\text{MSE}=\frac{1}{n}\sum(y_i-\hat y_i)^2\); \(\text{MAD}=\frac{1}{n}\sum\lvert y_i-\hat y_i\rvert\) (absolute); \(\text{RMSE}=\sqrt{\text{MSE}}\).
REC curve
The regression analogue of ROC: plots error tolerance (x) vs % of points within that tolerance (y) — it estimates the CDF of the error.
Extrapolation ≠ predicting the future

Olympic-100m men/women regression lines cross far in the future — nonsense. There is a real difference between predicting the future and blindly extrapolating beyond the training range. Always also check residuals, the sign of coefficients, and variables with extreme coefficients.

Cross-validation & tuning

Train / validation / test pipeline

  • Train builds the model; validation drives tuning / early stopping; test is touched once for the final evaluation.
  • Question: "how do we prevent lucky hits?" → cross-validation.

k-fold cross-validation

Split the training data into \(n\) folds. Train on \(n-1\) folds, validate on the held-out fold, and rotate so each fold is held out once.

⚠️ Exam watch-out — CV essentials
  • CV is for TUNING; the test set is for TRUE evaluation.
  • Compare configurations by the AVERAGE over folds, not the best single fold (avoids lucky hits). E.g. \(\lambda{=}0.01\) → avg 0.77 vs \(\lambda{=}0.02\) → avg 0.80 → pick 0.02.
  • Fit preprocessing per fold (on the train folds only) to avoid leakage.
  • CV is PESSIMISTICALLY biased (each fold trains on less data); LOOCV is approximately unbiased (folds differ from the full data by one point). Trap: "CV is optimistically biased" → false.

Variants

  • Nested / repeated CV — outer loop evaluates, inner loop tunes.
  • LOOCV — extreme: each fold leaves out a single instance.
  • Time-series CV — expanding window, test point just after train, sliding forward (cannot shuffle time).

Bias–variance

Decomposition \[ \text{Total Error} = \text{Bias}^2 + \text{Variance} + \text{irreducible} \]

As complexity rises, bias² falls and variance rises; total error is U-shaped with a minimum at the optimal complexity.

Validation is hard (leakage & over-reuse)

  • Feature engineering / transformations must be learned on the training set only — else leakage.
  • Even reserving the test set until the end, over-reusing the same train/validation split causes hidden overtraining — too many tuning runs and the test result disappoints.
  • "Scores too good to be true" → target leakage.
  • A test set is the only guarantee of generalization; not using one is out of the question.

Interpretability: opening the black box

White box vs black box

White box

Interpretable by design: linear/logistic regression, decision trees, decision rules, RuleFit, Bayesian. The easiest path to interpretability.

Black box

Tree ensembles, deep learning. High accuracy, low transparency → use agnostic techniques to explain importance, effects, and individual predictions.

Feature importance

MethodRetrain?Agnostic?Notes
MDI (impurity)NoNo (trees only)Biased; favours high-cardinality features; untrustworthy under overfit

Permutation importance — the mechanism

Shuffle one feature's values, predict again, and define importance = baseline performance − permuted performance. It captures interactions (shuffling breaks them), but tells you which features matter, not how much or in which direction. Use the test set for evaluation-style importance, the train set to learn what the model actually relied on.

⚠️ Exam watch-out — MDI is biased

MDI strongly favours high-cardinality (numerical) features and is untrustworthy under overfitting — a column of pure random numbers can outrank a real feature. Permutation / drop-column are model-agnostic alternatives.

PDP vs ICE

PDP (Partial Dependence)

Each point = the average prediction across all observations for a given feature value (others imputed / sampled). Shows the global, averaged effect.

ICE (Individual Conditional Expectation)

Same idea but one line per instance (no averaging) — reveals heterogeneity the PDP's average hides. The averaged ICE line = the PDP.

⚠️ Exam watch-out — PDP misses interactions

Because PDP holds everything else fixed (ceteris paribus), an interaction like \(y\sim x_0\cdot x_1\) gives flat-looking PDPs — the interaction does not show. Absence of evidence ≠ evidence of absence. Conversely, raw univariate data can imply false correlations under interactions — so inspect BOTH the data and the PDP.

LIME

LIME (Local Interpretable Model-agnostic Explanations)
Builds a local surrogate to explain a single prediction. Model-agnostic; works on tabular, text, and image data.
  1. Perturb: create samples around the instance \(x_i\) and have the black box predict them.
  2. Weigh by proximity to \(x_i\) (exponential kernel or \(k\) nearest neighbors).
  3. Fit a simple model — typically LASSO regression (regression because the output is the continuous probability; LASSO to keep the explanation sparse).
  • Text: perturb by randomly removing words. Images: perturb superpixels (groups of similar pixels, found via k-means) by blanking them — single pixels make no sense.
  • Pitfalls: defining the neighborhood/kernel and choosing the surrogate are arbitrary — the main reason LIME is less used now.

Shapley values & SHAP

Shapley value
From cooperative game theory (Lloyd Shapley): the fair payout per player, averaged over all orderings of the team. In ML, the fair contribution of a feature to a prediction.
Exact Shapley value for feature j \[ \varphi_j=\sum_{S\subseteq \{x_1,\dots,x_p\}\setminus\{x_j\}} \frac{|S|!\,(p-|S|-1)!}{p!}\,\big[v(S\cup\{x_j\})-v(S)\big] \]
Monte-Carlo approximation over M samples \[ \hat\varphi_j=\frac{1}{M}\sum_{m=1}^{M}\Big(\hat f(x^{m}_{+j})-\hat f(x^{m}_{-j})\Big) \]
Additivity — the explanation property

Start from the base-rate prediction and add each feature's contribution; they sum to the output. E.g. \(0.1 + 0.4 - 0.3 + 0.1 + 0.1 = 0.4\). SHAP is the most-used instance-explainability technique in industry.

⚠️ Exam watch-out — the Shapley trap

A Shapley value is the contribution of a feature to the given output for a given instance — NOT how much the output would change if you removed the feature. The SHAP dependence plot ≈ PDP (colour by a second feature for interactions); the force plot shows red/blue pushes around the base value.

What does an AUC of 0.5 mean?

  1. The model is 50% accurate at the default 0.5 threshold.
  2. Exactly half of the positives are classified correctly.
  3. A random positive is no more likely to outscore a random negative than chance — the model ranks no better than a coin flip.

One-minute recap

  • Generalization is everything. Split FIRST, stratify, prefer out-of-time; the test set catches failures and is used once.
  • Recall \(=TP/(TP{+}FN)\), precision \(=TP/(TP{+}FP)\), F1 = harmonic mean, balanced acc \(=(\text{recall}+\text{spec})/2\). All threshold-dependent.
  • Never trust accuracy under imbalance (99% by always predicting majority).
  • Sampling ⇔ inverse-cost matrix; resample TRAIN only; test accuracy usually drops; resampling often not worth it for ensembles / DL.
  • AUC = P(positive outscores negative), 0.5 = random, threshold-independent; PR beats ROC under imbalance; AR = 2·AUC − 1; lift of 1 = random; h-measure fixes AUC's cost-incoherence.
  • Adjusted r² can decrease; RMSE = √MSE; REC = regression ROC.
  • CV tunes, test evaluates; compare by fold average; CV is pessimistically biased, LOOCV ≈ unbiased; one-vs-all = \(k\), one-vs-one = \(k(k-1)/2\).
  • MDI biased; permutation = baseline − permuted (captures interactions); PDP averages & misses interactions, ICE = per-instance; LIME = local sparse surrogate; SHAP = contributions summing to the output (not a "removal effect").
