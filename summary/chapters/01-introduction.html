Introduction to Analytics & Big Data

The framing lecture: what data science really is, its three components and four quality criteria, the algorithm taxonomy, the process methodologies, the people — and big data, all viewed critically.

⏱️ ~18 min Components data · algorithm · purpose Criteria valid · useful · unexpected · understandable Methods CRISP-DM · SEMMA · Drivetrain Big Data the V's
◆ The one thesis to remember

Analytics is not (only) about machine learning or about big data. It is about extracting valid, useful, unexpected, understandable knowledge from data for a clear purpose — and that purpose is very often automation, not ML for its own sake.

What is data science?

Data science
The discipline of extracting value and knowledge from data. To unlock that value you must be able to store it, manage it, and analyze it.

Instead of hand-coding rules, you let the data speak — discover patterns and models from data to help predict, explain, or decide something.

◆ Synonyms — used loosely

Data Mining ≈ Big Data ≈ Data Analytics ≈ Data Science ≈ Knowledge Discovery ≈ AI ≈ Deep Learning ≈ Decision Science. The slide explicitly says: don't get too confused by Venn diagrams or infographics. Treating these as precise technical distinctions is a pitfall.

The three components & four quality criteria

◆ Why these matter

These are the minimal ingredients and the standard against which every result is judged — the spine of the whole course. No data = nothing to learn from; no algorithm = no learning; no purpose = no reason to act.

The three key components

Any data-science effort uses three things to discover patterns/models:

  • Data
  • An algorithm
  • A purpose

The four quality criteria

The discovered patterns/models must be:

CriterionMeans (one word)Threats / notes
ValidGeneralizableHolds on new data; threatened by time/seasonal effects, overfitting, sub-groups, regional differences. ML = generalizable correlation, not causation.
UsefulActionableMust map to a business question; consider implementation & maintenance cost, ease-of-use.
UnexpectedInterestingNon-obvious; balance trust vs discovery. Not all unexpected is interesting or valid.
UnderstandableInterpretableBlack box vs white box; trust, validity.
⚠ Exam watch-out

Match each criterion to its one-word synonym: Valid→generalizable, Useful→actionable, Unexpected→interesting, Understandable→interpretable. A frequent trap swaps these.

Valid: the "Tanks" / reward-hacking trap

◆ Spurious correlation

A "tank detector" learns the background/weather (tanks shot on sunny days, fields on cloudy days), not the tank — a non-generalizable correlation. Same flavor: a speed-rewarded self-driving RL agent learns to spin in circles; a Mars-mission planner optimizing total man-days survival proposes killing 2/3 of the crew. Key line: ML is about generalizable correlations, not necessarily causation.

Using data: structured vs unstructured

Structured (tabular / relational)

Rows × columns; ready for classical analytics.

Unstructured

Text, imagery, audio. The slide prefers naming the format: tabular, relational, text, imagery, audio.

Two routes to handle non-tabular data:

  1. Featurization — making it tabular.
  2. Use the data as-is with techniques that consume it directly — a strength of deep learning. Even then, some structure is still imposed (no truly structure-free modeling).

Basic terminology (tabular data)

Illustrated with the 14-row "Play Golf?" weather dataset (columns: Instance · Outlook · Temp · Humidity · Windy · Play Golf?).

Instances
Examples, rows, observations, customers, cases. Each row = one instance.
Instance identifier
A column to track instances (the 1–14 column).
Features (= attributes)
Input columns. Synonyms: fields, variables, predictors, covariates, explanatory variables, regressors, independent variables.
Target
Output column. Synonyms: label, class, dependent variable, response variable. May or may not be present.

Feature data types

  • Numeric (continuous) — e.g. Temp.
  • Categorical — discrete, nominal, ordinal, factor, binary (binary = special case of nominal).

Target data types

  • Numeric (continuous)
  • Binary
  • Multi-class
⚠ Exam watch-out

Binary is a special case of nominal categorical — not its own top-level type. And "feature = attribute = predictor = covariate = regressor = independent variable" all name the same column.

The algorithm taxonomy: supervised vs unsupervised

AspectSupervisedUnsupervised

GoalRelate predictors to target (\(X \to f \to Y\))Find structure / patterns in data

Also listed: semi-supervised, self-supervised, reinforcement learning.

Unsupervised sub-types

  • Clustering — construct groups over the dataset.
  • Association / sequence rule mining — find rules that describe the data.
  • Anomaly detection — find outliers.
  • Dimensionality reduction — reduce the number of features.
⚠ Exam watch-out

Unsupervised needs NO target. All four sub-types above (clustering, rule mining, anomaly detection, dimensionality reduction) are unsupervised — a classic true/false trap.

Supervised: classification vs regression

ClassificationRegression
Label typeCategoricalContinuous
Sub-typesBinary · Multiclass · Ordinal · MultilabelAbsolute values · Delta values · Quantile regression
⚠ Exam watch-out — bolded on the slide

"Binary classification forms the majority of applicative tabular settings." — memorize as TRUE.

Multiclass

One label per instance; more than 2 mutually exclusive outcomes.

Multilabel

Several labels can be true simultaneously; each outcome independently possible.

Ordinal classification (target is ordered categories) is not regression, even though the target has order. Single- and multi-output models are both possible.

Discriminative vs generative models

Discriminative

Directly model the input→label relationship as a function — i.e. \(P(\text{label}\mid\text{data})\) directly.

Generative

Model the joint probability density of data and labels; use Bayes' rule to get the label posterior (and can also generate data).

Bayes' rule (generative) \[ P(\text{label}\mid\text{data}) = \frac{P(\text{data} \,\&\, \text{label})}{P(\text{data})} = \frac{P(\text{data}\mid\text{label})\,P(\text{label})}{P(\text{data})} \]

Parametric vs non-parametric models

Parametric

Require specifying parameters / a function before predicting (e.g. linear regression assumes a linear form). Parameters then estimated via OLS, gradient descent, or other optimization.

Non-parametric (data-driven)

No explicit assumption about functional form — function approximation itself is part of learning. E.g. decision trees and ensembles.

⚠ Exam watch-out

"Non-parametric" does NOT mean "no parameters" — it means no fixed assumed functional form (the form is learned). Linear regression = parametric; decision trees = non-parametric.

Purpose: the five analytics goal types

◆ Why purpose is key

The crude equation "unsupervised = descriptive, supervised = predictive" is too simple. The slide stresses: a large majority of ML/AI use cases in business are not really about ML — they are about automation.

✓ Three quotable rules
  • Google Rule #1: "Don't be afraid to launch a product without machine learning." A heuristic gets you ~50% of the way if ML would give a 100% boost.
  • acolyer: increasing model performance does not necessarily translate into business value; changing the setup is a very effective way to unlock value.
  • Useful → actionable: "We can predict who will churn — and then what?" Don't build a model just to reverse-engineer an existing rule-based system.

Understandable & ethics

Understandable = humans can interpret the pattern (white box vs black box). Stakeholders ask: "Why does your model predict fraud for this case? Which attributes matter? What do you mean 'just trust us'?" A huge unreadable decision tree can be technically white-box yet practically uninterpretable.

◆ Ethics / fairness

Models rule credit scoring, employment, predicting recidivism. People gain a right to challenge a model's conclusion. References: Weapons of Math Destruction (Cathy O'Neil), ProPublica "Machine Bias", gender bias in word embeddings.

✓ "Weird small data" (Freakonomics)

Unusual but good-quality data can be valuable: a cookie-jar honesty system reveals holiday theft; gang bookkeeping explains dealers living with grandparents; UPS optimizes right turns; proper capitalization and Chrome/Firefox use correlate with good employees. Caveat: not everything unexpected is interesting or valid.

The analytics process model

⚠ Exam watch-out — know the exact grouping

Preprocessing covers Identify Business Problem → Transform the Data (steps 1–5); Analytics = Analyze (step 6); Post-processing = Interpret/Evaluate/Deploy (step 7). The neat linear diagram is idealized — the "real" version is a tangle of feedback loops (highly iterative / non-linear).

The people: data scientist & team

A data scientist should…

  • have solid quantitative skills
  • be a good programmer
  • excel in communication & visualization
  • have solid business understanding
  • be creative

Drew Conway Venn

  • Hacking ∩ Math/Stats = Machine Learning
  • Math/Stats ∩ Substantive = Traditional Research
  • Hacking ∩ Substantive (no stats) = Danger Zone!
  • All three = Data Science
◆ Key line

A data scientist is not a magic unicorn — a multidisciplinary team is needed (DB / lake-house admins & engineers, business expert, legal expert, data scientist, BI expert, data engineer, business enabler). Data engineers build/maintain architectures & pipelines; data scientists answer business questions, model, and tell stories.

Challenges & aftercare

Why projects fail (~60% never leave piloting)

  • Mapping the business question to a technique — no one-size-fits-all
  • Underestimating pre-processing effort
  • Too little training data; huge data imbalance or no labels
  • Data quality, noise
  • Predicting the future is hard (machines are naïve & lazy)
  • Office politics, expectation management

Aftercare (the "forgotten part")

  • Interpretation & validation by business experts
  • Sensitivity analysis (robustness to sample / assumptions / params)
  • Deploy into a business setting (user-friendly output)
  • Monitoring & backtesting (contrast output with observed)
◆ Stitch Fix insight

Data science teams over-focus on "model accuracy" (the easiest metric to measure), while business wants financial benefit, insight, interpretability. The goal of data science is not to execute — it is to learn. You learn as you go, not before you go.

Big Data & the V's

  • Volume — 90% of today's data created in the last 2 years; ~2.5 quintillion bytes/day.
  • Velocity — ~50,000 GB/second of global internet traffic.
  • Variety — ~90% of generated data is "unstructured"; 80% of growth is video/images/documents.
  • Veracity — 1 in 3 leaders distrust their data; poor quality costs the US ~$3.1 trillion/year.
  • Value — the fifth V.
⚠ Exam watch-out — both directions

You don't need big data to do analytics… but you can. Big data doesn't necessarily mean doing analytics… but it can. Big data and analytics are orthogonal.

One-minute recap

  • 3 components: data · algorithm · purpose. 4 criteria: valid (generalizable) · useful (actionable) · unexpected (interesting) · understandable (interpretable).
  • ML = generalizable correlation, not causation (Tanks trap). Most business ML is really automation.
  • Supervised needs a target (classification = categorical, regression = continuous); unsupervised needs none (clustering, rule mining, anomaly detection, dim. reduction).
  • Binary classification dominates tabular settings. Multiclass = one of >2 exclusive; multilabel = several at once.
  • Generative = joint density + Bayes' rule; discriminative = \(P(\text{label}\mid\text{data})\) directly. Non-parametric = no fixed functional form (trees), not "no parameters".
  • Process: Preprocessing (1–5) · Analytics (6) · Post-processing (7); really iterative. CRISP-DM (cyclic) · SEMMA · Drivetrain (Objective → Levers → Data → Models).
  • 4 V's: Volume, Velocity, Variety, Veracity (+ Value). Big data is orthogonal to analytics. AlexNet 2012 = deep-learning kickstarter.
