Big Data — MapReduce & Spark

From "is this even big data?" to HDFS, MapReduce, and Spark's in-memory DAG engine — only the exam-likely distinctions.

⏱️ ~18 min HDFS · MapReduce · YARN Spark · RDD · DataFrame · MLlib Streaming · Kafka · DStream · Structured Traps reduce associativity · Arrow vs Parquet

Big picture: most problems are not big-data problems

The deck is deliberately deflationary about big data: the Hadoop hype has passed, and you should pick the simplest tool that answers the business question. The escalation ladder is: in-memory → on-GPU / on-disk → distributed (Hadoop) → Spark → streaming.

◆ Why it matters

A 512 GB-RAM single server is often cheaper than an HDFS cluster, and on-disk libraries (Dask, Polars, DuckDB, vaex) handle data larger than RAM. "Pandas crashing on a 50 GB file is not an excuse." Insights > infra.

Big Data vs Analytics — orthogonal

Big data
How to manage / store huge, high-speed, unstructured data sets? (Does not imply you can do ML/AI/analytics on it.)
Analytics
How to analyze data and obtain insights? (Does not imply large volumes.)

These are independent axes. "Big data is anything which is crash Excel" is a joke, not a definition.

◆ The COST critique (McSherry et al.)

An unoptimised system can appear to "scale" better precisely because it was slow to begin with. A single well-written thread can outperform a big cluster. Be honest about whether you actually have big data.

Hadoop: the three core pieces

Genesis: Google File System paper (2003) → Google MapReduce paper. Hadoop grew out of the Nutch search engine (Doug Cutting & Mike Cafarella); split off at Yahoo! in 2006, open-sourced 2008, named after a toy elephant. Now under the non-profit Apache Software Foundation.

Four core modules

  • Hadoop Common — shared libraries.
  • HDFS — distributed storage (Java-based).
  • MapReduce — parallel computation model.
  • YARN (Yet Another Resource Negotiator) — resource scheduling.

HDFS — storage only

Name node
Makes all decisions about block replication (the "brain" / metadata).
Data nodes
Hold the actual data blocks.
  • Typical block size = 64 MB; large files = multiple 64 MB blocks, each placed on separate data nodes.
  • Replication gives fault tolerance; replica count is set at file-creation time.
  • "A massive, distributed C: drive." Storage, not yet computing — naively reading a huge file locally still ends badly.

YARN & Hadoop versions

  • Hadoop 1: the JobTracker managed the cluster — it didn't scale.
  • Hadoop 2: split MapReduce into YARN (resource management) + MapReduce (compute). Single active + single standby NameNode.
  • Hadoop 3: erasure coding — 1 parity block per 2 data blocks → only 1.5× disk space (vs 3× with replication in Hadoop 2), same fault tolerance; supports multiple standby NameNodes.
⚠ Exam watch-out

HDFS = storage; MapReduce = computation; YARN = resources. Don't swap them. Erasure coding's 1.5× is a favourite T/F number. The name node decides replication; data nodes only hold blocks.

MapReduce: map → shuffle → reduce

A programming framework for coordinating fault-tolerant, scalable tasks over (key, value) pairs. HDFS stores files distributed; MapReduce does something with them distributed.

Functional roots

Map

Apply a function to every item → a new list.

\[ [1,2,3,4,5] \xrightarrow{x\,\mapsto\,x^{2}} [1,4,9,16,25] \]

Reduce

Aggregate a list → a scalar.

\[ [1,2,3,4,5] \xrightarrow{\text{sum}} 15 \]

The Hadoop pipeline

  1. Map — each input pair → zero, one, or more output pairs. Runs in parallel.
  2. Shuffle / distribute — entries with the same key go to the same worker.
  3. Reduce — one reduce function per key group → final pair per distinct key; output optionally sorted per key.

The reduce-associativity trap (CRITICAL)

Reduce runs on partial results, possibly multiple times in a chain (you can start reducing before all mappers finish). So reduce must satisfy two requirements:

  1. Emit the same structure as the map output (so it can be re-reduced).
  2. Give correct results even when applied multiple times on partial results.

Naive averaging — WRONG

reduce yields sum(values)/len(values). drama→[100,200] gives 150. A slow mapper later adds drama→300; re-reducing [150,300] gives

\[ \frac{150+300}{2} = 225 \neq 200 \]

The true mean of \(100,200,300\) is \(200\). Uh-oh!

Correct — carry (avg, count)

Map yields (genre, (nrPages, 1)); reduce computes a weighted running average:

\[ \text{avg} = \frac{\sum_{i} \text{avg}_{i}\, n_{i}}{\sum_{i} n_{i}} \]

[(100,1),(200,1)]→(150,2); then with (300,1): (150·2+300·1)/(2+1)=(200,3). Correct!

SQL on Hadoop & why it disappointed

"From the moment a distributed store gets popular, the next question is how to run SQL on it." The honest answer: SQL-on-Hadoop is not a perfect RDBMS — it sacrifices speed, SQL compatibility, and complex joins.

◆ Where it landed

"Big Data" interest peaked 2011–2014; in 2015 the crowd moved to AI/ML/deep learning. Today storage/querying stabilised around cloud warehouses (Redshift, Snowflake, Clickhouse, Presto, Dremio). "Don't do it yourself." MapReduce was never well suited for analytics (slow HDFS in-out swapping — ask the Mahout project). "2015 was the year of Apache Spark."

Spark: in-memory DAG engine that kicks out MapReduce

Apache Spark = real-time, in-memory, parallelized data processing with its own SQL engine. Benchmarks: 10–100× faster than Hadoop MapReduce.

◆ What Spark keeps vs throws out

Keeps: a resource manager (YARN, but also Mesos or standalone) and storage (HDFS, S3, Cassandra, HBase, Hive, flat files). Throws out: MapReduce. It replaces it with an advanced DAG execution engine supporting cyclic data flow and in-memory computing (note the similarity to Dask).

Building blocks

Base = Spark Core Engine. On top: Spark SQL, Spark Streaming, MLlib (ML), GraphX (graphs), Spark R; above Spark SQL sits BlinkDB (approximate SQL).

RDD — Resilient Distributed Dataset

RDD
Spark Core's primary data abstraction: an immutable collection of elements partitioned across the cluster, operated on in parallel.
Fault tolerance
By tracking the lineage of transformations — the chain RDD1→RDDn is logged and re-computed on node failure.
Efficiency
Parallelization + minimizing data replication; RDDs stay in memory where possible (great for iterative queries).

Transformations lazy

Create a new RDD (original unchanged). Not executed until an action needs them.
map, filter, flatMap, union, distinct, reduceByKey, groupByKey, join, sortByKey

Actions eager

Produce a value / side-effect; trigger execution. Measure, don't change the data.
reduce, count, first, take, collect, saveAsTextFile, foreach

⚠ Exam watch-out

Transformations are lazy; actions are eager. Fault tolerance = lineage re-computation, NOT replication of the data itself. RDDs are immutable — a transformation never mutates the original.

Execution: DAG → stages → tasks

Writing Spark / PySpark

  • RDD apps can be coded in Java or Scala (JAR), or interactively via the Spark shell; APIs in Scala/Java/Python/R/SQL.
  • The driver program sets up a SparkContext = connection to the cluster. Operations are NOT executed on your machine — they're sent to the cluster. No data is loaded in the client... unless you call .toPandas().
counts = text_file.flatMap(lambda line: line.split(" ")) \
                  .map(lambda word: (word, 1)) \
                  .reduceByKey(lambda a, b: a + b)

SparkSQL, DataFrames & Datasets

RDDs still "feel like MapReduce" and are very low-level. SparkSQL builds on Spark Core to load/query structured data via SQL, HiveQL, or a Pandas-like API. At its core is the Catalyst query optimizer.

◆ Since Spark 2.0

Spark SQL is the primary, feature-rich interface, hiding RDDs behind higher-level abstractions. You use a SparkSession (not a SparkContext).

⚠ Exam watch-out

DataFrame = Dataset[Row] (untyped). Dataset[T] adds compile-time type safety but exists only in Scala/Java — Python & R have no compile-time typing, so they get DataFrames only. A DataFrame imposes named-column structure on top of RDDs and can be converted back to an RDD.

people.json mini-computation (very testable)

Rows: (age=null, Michael), (30, Andy), (19, Justin).

  • df.filter(df['age'] > 21) → only (30, Andy) (null fails the comparison).
  • df.select(df['name'], df['age']+1) → Michael→null, Andy→31, Justin→20.
  • df.groupBy("age").count() → 19:1, null:1, 30:1.

MLlib — "scikit-learn on Spark"

  • Provides ML algorithms, Featurization, Pipelines, Persistence, Utilities.
  • Since Spark 2.0 the primary API is the DataFrame-based spark.ml; the older RDD-based spark.mllib reached feature parity in 2.3 and is in maintenance mode in Spark 3.

Streaming analytics

Beyond static models: act as components of real-time feedback loops. But "streaming analytics might not be that hard" — and many use cases are really just personalization.

Three aspects of "streaming" (don't conflate)

Streaming data
The source emits a stream — store it, access history, ship it around. Mostly plumbing.
Streaming training
On-line algorithms trained on a continuous stream — very hard, ad-hoc per algorithm.
Streaming prediction
Predict on a stream — a matter of deployment.
◆ The reality on online algorithms

Usually you don't need online training. Even Netflix (9M events/sec) does main training offline. Pattern: handle streaming data + online prediction, but train offline. Re-train cadence depends on how fast the feature space changes, not on how fast data arrives.

Bounded vs unbounded; windows

  • Data: bounded (finite) or unbounded (infinite). Engine: streaming or batch. Combinations are possible.
  • Finite: simple — reprocess on failure, use checkpoints (the "resilient" in RDD). Infinite: must account for time (event creation / ingestion / processing); aggregate over a WINDOW.
  • Sliding window: overlapping, slides by an interval (time- or count-based). Hopping (tumbling): non-overlapping consecutive windows.

The pipeline & Kafka

Producers → Collector (Flume, NiFi) → Broker (Kafka, RabbitMQ) → Processor (Flink, Spark, Storm) → Indexer (ElasticSearch) → Visualizer (Kibana).

  • Kafka = popular event broker = message-oriented middleware that decouples apps; easily integrated with Spark. But "in itself, not much analytics."
  • Event processors are where the actual analytics happen.

Spark Streaming vs Structured Streaming

Spark Streaming (DStream)Structured Streaming
Built onSpark Core (RDDs)Spark SQL engine
AbstractionDStream = sequence of RDDsunbounded table (continuously appended)
Modelmicro-batchingincremental query, trigger-based; batch-like code
Statefullminimal — does NOT materialize the whole table; discards source

⚠ Exam watch-out

DStream = a sequence/series of RDDs (each holding one interval); Spark Streaming runs in micro-batch mode. Structured Streaming treats the stream as an unbounded table, runs incrementally, does not materialize the full table, and correctly updates old windows with late data (an event generated 12:04 but arriving 12:11 still increments the 12:00–12:10 window).

  • Flink = true streaming (not micro-batch) — used where mini-batching isn't viable.
  • Storm = "the Hadoop for real-time"; mostly replaced by Kafka + Spark.
  • Beam = unified model for data-parallel pipelines.

MLOps in one screen

  • Sculley et al. (2015), "Hidden technical debt": only a small fraction of a real ML system is actual ML code.
  • Models degrade silently via concept drift — solid model governance is key.
  • Breck et al. (2016), "ML Test Score": set up common monitoring/logging.
  • Push vs Pull: push = run on a batch, save outputs to the data layer; pull = deploy as an API/microservice.
  • Canary/shadow deployment: new model runs silently alongside the current one. Industry mostly settled on Docker. Don't split the data layer into raw/processed/final.

One-minute recap

  • Big data ≠ analytics — orthogonal; one doesn't imply the other. In-memory is often enough.
  • Hadoop: Common, HDFS (storage; 64MB blocks; name node decides replication), MapReduce (compute), YARN (resources). Hadoop 3 erasure coding = 1.5× vs 3× replication.
  • MapReduce: map (0/1/many pairs) → shuffle (same key→same worker) → reduce (scalar). Reduce must emit map's structure & be correct on partial results → carry (avg, count) for averaging.
  • Hive compiles SQL→MapReduce (slow); HBase = first DB, No SQL; Impala/Tez drop MapReduce.
  • Spark kicks out MapReduce for a DAG, in-memory engine (10–100×). RDD: immutable, lineage fault tolerance, lazy transformations / eager actions.
  • SparkSQL (Catalyst) primary since 2.0 via SparkSession. DataFrame = Dataset[Row] (untyped); Dataset[T] typed, Scala/Java only. MLlib spark.ml primary since 2.0.
  • Formats: AVRO=row/ETL, Parquet=column/query, Arrow=in-memory spec (not a file), Feather on Arrow, HDF5 for DL weights.
  • Streaming: 3 aspects (data/training/prediction); online training rarely needed. DStream = micro-batch sequence of RDDs; Structured Streaming = unbounded table, handles late data.
