1

Weights of Evidence (WoE) encoding and Information Value (IV) are used to encode and screen a high-cardinality categorical predictor against a binary target.

Ch. 02 · Preprocessing

a
Define WoE for a category and explain the meaning of its sign. State the property that makes WoE attractive for use in a linear/logistic model.
Model answer
  • WoE for a category is \( \text{WoE}_{cat} = \ln\left(\frac{p_{c1,cat}}{p_{c2,cat}}\right) \), where \( p_{c1,cat} \) is the share of all class-1 instances that fall in the category and \( p_{c2,cat} \) the share of all class-2 instances that fall in it.
  • Sign: if \( p_{c1,cat} \gt p_{c2,cat} \) the WoE is positive; if \( p_{c1,cat} \lt p_{c2,cat} \) it is negative. It contrasts how over- or under-represented a category is for one class versus the other.
  • Key property: WoE has a monotonic relationship with the target (e.g. higher WoE = less risk). Replacing the category by its single WoE number gives one numeric feature already aligned monotonically with the target, which suits linear/logistic models and avoids many dummy columns.
b
Information Value is \( IV = \sum (p_G - p_B)\cdot \text{WoE} \). Explain how IV is used in practice, and state why both WoE and IV must be computed only after the train/test split.
Model answer
  • IV aggregates the WoE contributions across all categories of a variable into a single score of how well the variable separates the two classes; it is used to screen variables, keeping those with \( IV \gt 0.1 \) as predictive.
  • One can also optimize the category cut-points to maximize IV, and run a sensitivity analysis of IV versus the number of categories to choose a parsimonious binning at the elbow.
  • Both WoE and IV are computed from the target (class counts per category), so computing them on the full data before splitting leaks target information from the held-out instances into the features. They must be fit on the training data only and then re-applied to validation/test/production, like any other pipeline statistic.
2

Bagging and boosting are the two dominant ensemble paradigms for trees.

Ch. 04 · Trees & Ensembles

a
Compare bagging and boosting along (i) how base models are constructed, (ii) whether bootstrapping is used, and (iii) the core intuition behind each.
Model answer
  • Construction: bagging builds base models independently and in parallel, each on its own resampled data; boosting builds models sequentially, each new learner targeting the errors of the current ensemble.
  • Bootstrapping: bagging draws bootstrap replicas (sampling with replacement) for each model; boosting does not bootstrap, it reweights instances so that misclassified ones get heavier weight in the next round.
  • Core intuition: bagging adds diversity so that independent errors cancel under aggregation; boosting relies on cooperation, each learner correcting what previous ones got wrong.
b
Using the bias-variance decomposition, explain which type of base model each method is best suited to and which error component each mainly reduces.
Model answer
  • Bagging mainly reduces variance, not necessarily bias. By averaging many diverse predictions it shrinks the variance term while leaving bias roughly unchanged, so it suits high-variance, low-bias (complex) base learners such as deep decision trees (e.g. random forests).
  • Boosting mainly reduces bias, not necessarily variance. By sequentially correcting systematic mistakes it lowers the bias term, so it suits low-variance, high-bias (simple/weak) base learners such as shallow trees (e.g. AdaBoost, GBM, XGBoost).
  • Caveat: boosting can overfit if run too long (overboosting), so regularization / limiting rounds matters; bagging is comparatively robust, adding more trees does not degrade validation performance.

Apriori frequent-itemset mining and the downward-closure property.

Ch. 06 · Unsupervised

a
Explain the two-stage structure of the apriori algorithm and state, with the power-set argument, why a brute-force search for frequent itemsets is infeasible for many items.
Model answer
  • Apriori is a two-stage algorithm. Stage 1: apply a minimum support threshold to find all frequent itemsets. Stage 2: apply a minimum confidence threshold to those frequent itemsets to form frequent association rules. Support comes first, confidence second.
  • Brute force would require evaluating every possible itemset. The set of possible itemsets is the power set over the item set \( I \), of size \( 2^{|I|} - 1 \) (excluding the empty set).
  • This grows exponentially: for 6 items there are 63 itemsets, but for 100 items it is already \( 2^{100} - 1 \) (on the order of \( 10^{30} \)), so enumerating all of them is computationally infeasible.
b
State the downward-closure (anti-monotonicity) property of support and explain how the 'join and prune' candidate-generation step exploits it.
Model answer
  • Downward closure / anti-monotonicity: for a frequent itemset, all of its subsets are also frequent; equivalently, for an infrequent itemset, all of its supersets must also be infrequent.
  • This lets apriori avoid enumerating the full power set: once an itemset is below the support threshold, none of its supersets can be frequent, so they are never generated.
  • Join step: to build candidate k-itemsets, join the set of frequent (k-1)-itemsets with itself to produce length-k candidates.
  • Prune step: discard any candidate that contains a (k-1)-subset which was not frequent, since by anti-monotonicity such a candidate cannot be frequent. Only the surviving candidates have their support counted, and the process repeats per size level.

MapReduce processes data as a pipeline over (key, value) pairs.

Ch. 08 · Big Data

a
Explain the three stages of the Hadoop MapReduce pipeline (map, shuffle/distribute, reduce). For each stage state what it does and one key property (e.g. parallelism, how many outputs map may emit, what determines where a value is sent).
Model answer
  • Map: applies a function to each input (key, value) pair and emits a list of output (key, value) pairs. It may emit zero, one, or more pairs per input. Map tasks run in parallel over the input pairs, so the input partitioning drives the parallelism.
  • Shuffle / distribute: the intermediate pairs emitted by the mappers are grouped and routed so that all entries sharing the same key are sent to the same reduce worker. The key (not the value) determines the destination worker; this exchange of intermediate pairs across nodes is the shuffle.
  • Reduce: each worker receives all values for a given key as a group and applies a reduce function to produce a final (key, value) pair per distinct key. Final outputs are written to HDFS (e.g. part-r-00000) and are optionally sorted per key.
b
The deck states a reduce function must (1) emit the same structure that map emits and (2) be correct even when applied multiple times on partial results. Explain WHY both requirements matter, using the idea of stragglers / partial reduction.
Model answer
  • Reduce should be able to run on partial results and be chained: reducing a complete list at once may be too large, you may want to start reducing before all mappers finish, and you keep reducing until one item per key remains.
  • Requirement (1) same structure: if reduce emits the same shape as map, its output can itself be fed back into reduce. If the output looked different from the map output, a second reduce pass over partial results would be impossible.
  • Requirement (2) correctness on partials: because a slow mapper (straggler) can produce extra values after an earlier partial reduction, reduce may be re-applied to a mix of partial results and fresh values. The function must give the same final answer regardless of how the inputs were grouped into partial reductions.
  • Averaging illustrates the trap: carrying only a running average violates (2), because re-reducing [150, 300] gives 225 instead of the true mean 200. Carrying (running average, count) restores correctness, since the weighted recombination \(\frac{150\cdot 2 + 300\cdot 1}{2+1}=200\) is right and the (avg, count) output matches the map structure.
